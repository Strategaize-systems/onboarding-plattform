## Problems found

- **AC #2 nur partiell live-verifiziert.** Die RPC-Smoke deckt aktuell nur den NULL-Default-Pfad ab (`rpc_get_branding_for_tenant(NULL)` -> Strategaize-Default-JSON). Die Pfade fuer `partner_client mit Parent`, `partner_organization eigene` und `direct_client mit Default-Payload` sind code-side via Mock-Vitest abgedeckt (Cases 2/3/4 in resolve.test.ts), aber nicht gegen die Live-DB getestet. MT-10 (`rpc-branding.test.ts` gegen Coolify-DB mit 16 Faellen via SAVEPOINT-Pattern) holt das nach.
- **Migration 091 DDL inside DO-Block:** Das `DO $mig034_step2$ BEGIN ... END $mig034_step2$;` Pattern enthaelt direkt DDL (CREATE TABLE, ALTER TABLE, CREATE INDEX, CREATE POLICY, GRANT) sowie eine eingebettete `CREATE OR REPLACE FUNCTION` mit eigenem `$func$`-Tag. Pattern-Reuse von Migration 084 + 089 — keine bekannten Limitierungen, aber die `IF NOT EXISTS (...) THEN ... END IF;` Constraint-Lookups + die 2 nested `DECLARE ... BEGIN ... END;` Sub-Bloecke fuer die CHECK-Erweiterungen sind eine ungewohnte Mischform. Apply-PASS bestaetigt, dass das funktioniert; bei zukuenftigen Migrations koennte man den Top-Level auf reines `BEGIN; ... COMMIT;` aendern (siehe Migration 090) und nur die DECLARE-Bloecke als `DO $$ ... $$` schreiben — stilistisch sauberer.
- **Backfill 0/0 Partner-Tenants:** Production hat noch keine `tenant_kind='partner_organization'`-Tenants, daher hat der Backfill nichts eingefuegt. Das ist erwartet (V6 ist Pre-Pilot), aber bedeutet, dass die Backfill-Logik selbst nicht live-verifiziert ist. Die Logik ist trivial (`INSERT ... SELECT ... ON CONFLICT DO NOTHING`), aber bei der ersten echten Partner-Anlage muss validiert werden, dass der Backfill korrekt griff (alternativ: SLC-102 Server Action `createPartnerOrganization` legt die Branding-Row direkt mit an — Code-Drift-Risiko).
- **CHECK-Erweiterungen sind "noisy idempotent":** Re-Run der Migration produziert DROP+RECREATE der `validation_layer_reviewer_role_check` + `block_checkpoint_checkpoint_type_check`, jedes Mal mit NOTICE-Output. Endzustand identisch, aber funktional ist ein eleganterer "skip wenn schon enthaelt"-Check via `pg_get_constraintdef`-Vergleich denkbar. Akzeptiert per Pattern-Reuse aus Migration 040.
