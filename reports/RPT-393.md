#### L-3: Partial-Bulk-Update zeigt stale UI bis User-F5 (neue Beobachtung in MT-3)

`updateEmailClassifications` iteriert sequenziell ueber das updates-Array:

```typescript
for (const u of updates) {
  const { error: updateError, count } = await supabase.from(...).update(...).eq(...);
  if (updateError) {
    return { ok: false, error: `UPDATE fehlgeschlagen fuer ${u.message_id}: ${updateError.message}` };
  }
  updatedCount += count ?? 0;
}
```

Wenn Bulk-Reclassify-Update Nr. 26 von 50 fehlschlaegt (z.B. RLS-Reject auf eine spezielle Row, Network-Timeout), sind die ersten 25 in der DB persistiert + die Action returnt `{ ok: false, error: ... }`. Der Client zeigt die Error-Message, ruft `router.refresh()` NICHT (steht nur im success-Path). Die UI zeigt weiter die alten 50 Labels. Erst nach manueller Seiten-Aktualisierung (F5) sieht der User: 25 mit neuen Labels + "manuell"-Badge, 25 unveraendert.

**Auswirkung:** UX-Verwirrung ("Was wurde aktualisiert?"), aber **keine Daten-Korrektheit-Probleme** — die Datenbank ist konsistent, jede einzelne UPDATE ist atomar mit `pre_filter_corrected=true`.

**Schweregrad:** Low — V9.0-Test-Corpus ist <500 Emails, Real-World-Partial-Failure-Rate ist niedrig (RLS deterministisch, Same-Tenant-Network reliable). F5 fixt es.

**Empfehlung:** Optional in MT-8 Records-Cleanup: auch im Error-Path `router.refresh()` + Error-Message in der Toast-Area. Alternative-Loesung waere RPC-batch-Update, aber das geht ueber MT-3-Scope.
