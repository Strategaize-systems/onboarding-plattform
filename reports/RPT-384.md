## Findings

### High

#### F-1 — Next.js Server-Action `bodySizeLimit` nicht konfiguriert
- **Location**: `next.config.ts` enthaelt keinen `experimental.serverActions.bodySizeLimit`-Eintrag
- **Impact**: Next.js Default-Limit fuer Server-Action-Payloads = **1 MB**. UI akzeptiert Files bis 500 MB (`MAX_FILE_SIZE_BYTES = 524288000` in helpers.ts) und Storage-Bucket hard-capt ebenfalls auf 500 MB (Migration 106 Line 475). Aber `uploadBulkEmailRun` wird **nie aufgerufen** wenn der File > 1 MB ist — Next.js wirft `Body exceeded 1mb limit` an der HTTP-Pipeline, BEVOR der Server-Action-Handler laeuft.
- **Real-World-Impact**: Gmail-Takeout-`.mbox` ist typisch 100 MB - mehrere GB. Einzelne `.eml` sind oft 2-5 MB (HTML-Mails + Inline-Images). **Alle echten Bulk-Email-Upload-Faelle scheitern derzeit** — nur synthetische Mini-Files <1 MB funktionieren.
- **AC-Bruch**: AC-SLC-165-2 (.mbox-Upload) + AC-SLC-165-3 (Multi-.eml-Upload) sind in Real-World nicht erfuellt
- **Warum nicht durch Vitest erfasst**: Aktuelle 17 Tests mocken `@/lib/supabase/server` + `@/lib/supabase/admin` und rufen `uploadBulkEmailRun(formData)` direkt auf. Sie umgehen Next.js' HTTP-Body-Pipeline. Vitest pruefen die Action-Logik, nicht die Server-Action-Transport-Schicht.
- **Fix**: 1-Block-Erweiterung in `next.config.ts`:
  ```ts
  const nextConfig: NextConfig = {
    output: "standalone",
    serverExternalPackages: ["pdf-parse", "mammoth"],
    experimental: {
      serverActions: {
        bodySizeLimit: "500mb",
      },
    },
    // ... rewrites, headers ...
  };
  ```
- **Caveat**: 500 MB Body-Buffering in Next.js belastet App-Container-RAM. Bei mehreren parallelen Uploads moeglich OOM. Vermutlich vertretbar fuer V9.0 (Single-User-Upload, niedriger Concurrency-Druck), aber als ISSUE fuer V9.1+ vermerken (Stream-Threshold + chunked-Upload).
- **Severity**: High — Slice nicht real-world-funktional ohne Fix
- **Aktion**: ISSUE-087 in `docs/KNOWN_ISSUES.md` angelegt, Pflicht-Fix vor `/backend SLC-165 MT-5`

### Low

#### F-2 — Race-Window im Duplicate-Check Pre-Check + INSERT
- **Location**: `actions.ts:198-212` (SELECT Pre-Check) + `actions.ts:235-256` (INSERT mit UNIQUE-Backstop)
- **Impact**: Zwei zeitgleiche Uploads identischer Datei (gleicher `file_hash`, gleicher `tenant_id`): beide passieren Pre-Check, einer INSERTet erfolgreich, der zweite trifft auf UNIQUE-Constraint. Zweiter Caller erhaelt kryptischen `email_bulk_run INSERT fehlgeschlagen: 23505 duplicate key` statt sauberem `{ ok: true, duplicate: true }`.
- **Tatsaechliche Wahrscheinlichkeit**: Sehr gering. V9.0 = single-uploader pro Tenant, Race-Window ist max einige hundert ms zwischen Pre-Check und INSERT.
- **Akzeptanz**: Im Code dokumentiert ("UNIQUE-Constraint backstop"). Slice-Spec verlangt keine Race-Condition-Behandlung. Storage-PUT mit `upsert: false` blockiert ebenfalls als 2. Backstop (`storagePath` deterministisch). Sauberer Fix waere ein DB-Layer-Pattern (INSERT ... ON CONFLICT DO UPDATE RETURNING id), aber out-of-MT-4-Scope.
- **Severity**: Low — Edge-Case ohne user-impacting Folge

#### F-3 — `getBulkRunById` ohne expliziten `tenant_id`-Filter
- **Location**: `actions.ts:311-329`
- **Impact**: Function nutzt `createClient` (user-context) und filtert nur per `.eq("id", runId)`. Verlaesst sich exklusiv auf RLS-Policy `email_bulk_run_tenant_select` fuer Tenant-Isolation. Bei einer RLS-Disabled-Misconfiguration (sehr unwahrscheinlich) wuerde Cross-Tenant-Lookup moeglich. Defense-in-Depth-Hinweis.
- **Akzeptanz**: Die Funktion ist explizit Pre-Declaration fuer MT-6, dort wird der reale Use-Case gebaut. Standard-Pattern im Repo nutzt RLS allein. Kein Tenant-`.eq("tenant_id", ...)` Filter in Sister-Funktionen.
- **Severity**: Low — Hinweis, keine Aktion erforderlich
