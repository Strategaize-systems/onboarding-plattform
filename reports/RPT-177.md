# /doctor SLC-075 Round 3 — Bucket-MIME-Filter vs codec-Suffix

## Problem Summary

Browser-Smoke Round 3 nach Commit `6f24b5c`:
- **AC-10/AC-11 PASS** (Sidebar + Self-Spawn + MediaRecorder)
- **AC-12 FAIL Round 3**: Upload schlaegt mit `HTTP 400 {"statusCode":"415","error":"invalid_mime_type","message":"mime type video/webm;codecs=vp9,opus is not supported"}` fehl

Diff zu Round 2 (HTTP 401): jetzt erreicht der PUT Kong + Storage, scheitert an Bucket-MIME-Filter.

## Observed Symptoms

User-Screenshot Round 3:
- URL: `/employee/walkthroughs/33ea58be.../record`
- Inline-Fehler: `Fehler: Upload-Fehler: Upload fehlgeschlagen mit HTTP 400 { "statusCode":"415", "error":"invalid_mime_type", "message":"mime type video/webm;codecs=vp9,opus is not supported" }`

## Most Likely Cause

`sql/migrations/084_v5_walkthrough_storage_bucket.sql` legt den Bucket mit:
```sql
allowed_mime_types = ARRAY['video/webm']
```
an — das ist exact-match, **keine Prefix-Suche**. MediaRecorder gibt aber den vollen Codec-String zurueck, das `pickMimeType()`-Helper liefert `video/webm;codecs=vp9,opus` (preferred) oder `video/webm;codecs=vp8,opus` (fallback). Der `XMLHttpRequest.setRequestHeader("Content-Type", mimeType)` schickt den vollen String → Storage-Bucket-Filter rejected.

RFC 7231 §3.1.1.5 erlaubt das Strippen des `;codecs=...`-Parameters — `video/webm` allein ist ein gueltiger Content-Type. Der Stream-Inhalt bleibt unveraendert (das Blob hat weiter den vollen MIME-Type), und Browser-Detection bei spaeterem Download nutzt blob-content-Sniffing, nicht den ursprunglichen Header.

## Why Not a SQL-Migration

Alternative-Fix: `UPDATE storage.buckets SET allowed_mime_types = NULL WHERE id='walkthroughs'`. Diskutiert + verworfen:
- `NULL` setzt _alle_ MIME-Types erlaubt — verliert Schutz gegen versehentliche Uploads von Bildern, PDFs, etc.
- `ARRAY['video/webm', 'video/webm;codecs=vp9,opus', 'video/webm;codecs=vp8,opus']` waere brittle (jeder neuer Browser mit eigenem Codec-String braeche das wieder)
- Client-side Header-Strip behebt das Symptom an der richtigen Stelle: HTTP-Header-Compliance mit RFC 7231
