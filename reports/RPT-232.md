## Implementation-Details

### Server Actions (`actions.ts`)

Zwei separate Server Actions, beide partner_admin-only mit Defense-in-Depth:

1. **`uploadLogo(formData: FormData)`** — File-Upload nach Storage-Bucket
   - File-Validation (Server-side authoritative): `file instanceof File`, `file.size > 0 && <= 524288`, `file.type in ['image/png','image/svg+xml','image/jpeg']`
   - Inline-Auth-Check via `requirePartnerAdmin()` (User existiert + role='partner_admin' + tenant_id)
   - Storage-Pfad erzwungen aus `tenantId` + MIME-abgeleiteter Extension: `${tenantId}/logo.${ext}` (kein User-Input bestimmt den Folder)
   - `admin.storage.from('partner-branding-assets').upload(path, arrayBuffer, { contentType, upsert: true })`
   - DB-Sync via `upsertBrandingPath(...)` Helper (UPDATE → bei rowCount=0 → INSERT) — deckt Edge-Case "Backfill-Row fehlt" ab
   - Audit via `captureInfo` mit `metadata.category = 'partner_branding_logo_updated'`
   - `revalidatePath('/partner/dashboard/branding')` + `revalidatePath('/partner/dashboard')`

2. **`updateBranding(formData: FormData)`** — Color + Display-Name
   - Hex-Validation serverseitig: `HEX_REGEX = /^#[0-9a-fA-F]{6}$/` fuer primary_color (required) und secondary_color (optional, leer oder Hex)
   - Inline-Auth-Check
   - `upsertBrandingPath(...)` mit `{ primary_color, secondary_color, display_name }`
   - Audit via `captureInfo` mit `metadata.category = 'partner_branding_updated'`

`upsertBrandingPath` ist ein interner Helper, der UPDATE bevorzugt (Backfill aus Migration 091 hat Row angelegt) und auf INSERT zurueckfaellt nur wenn rowCount=0. Defense-in-Depth: Tenant-Filter ist im UPDATE-WHERE und im INSERT-Body, RLS-Policies pbc_update_own_partner_admin / pbc_insert_own_partner_admin pruefen dasselbe auf DB-Ebene.

## Problems found

**Keine Blocker. Vier akzeptierte Befunde:**

1. **L-1 Size/Mime-Validation nicht im DB-Layer testbar.** Die FormData-File-Pruefung lebt in der Server-Action und kann nicht durch direkten pg-Client-INSERT geprueft werden, ohne Storage-Engine-spezifische Annahmen zu treffen. Wird ueber Browser-Smoke MT-13 abgedeckt. Akzeptabler Tradeoff der DB-Test-Strategie aus MT-5.
2. **L-2 Page-Layout abweichend von Spec.** Slice-Spec sagt "3 Sektionen (Logo + Color + Vorschau-Frame)" als Cards. Implementiert ist EIN Card mit Editor-Component, der intern Logo-Form + Color-Form + Live-Preview in Grid-Layout (lg:grid-cols-2) anordnet. Aequivalent in UX (alle Sektionen sichtbar, Live-Preview reagiert), kompakter im Code. Visual-Smoke MT-13 muss bestaetigen.
3. **L-3 Logo-Cleanup beim MIME-Wechsel nicht implementiert.** Bei Wechsel PNG → SVG bleibt die alte logo.png als Waise im Bucket. Storage-Bucket ist klein (max 500 KiB pro File), kein Production-Risk. Slice-Spec hat keinen Cleanup gefordert — V6.1+ falls Bucket-Lifecycle benoetigt wird. Werde im Cockpit-Backlog als BL-XXX-Kandidat erwaehnen.
4. **L-4 BrandingPreview ist Mockup, nicht Live-iframe.** Slice-Spec sagt "iframe oder eingebettete Component". Wir haben Mockup gewaehlt — Mandanten-Auth-Session im iframe ist kompliziert und verfaelscht das Resultat. Mockup zeigt Branding-Aenderungen sofort vor DB-Write. Funktional aequivalent fuer Live-Preview-Anspruch.
