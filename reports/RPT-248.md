   `finalized` capture_session des Mandanten + prueft `lead_push_consent` —
   gibt `captureSessionId | null` an `<PartnerClientWelcomeBlock>` weiter.
   - Admin-Client-Lookup gerechtfertigt (deterministische Index-Query, RLS
     erlaubt Mandant zwar SELECT auf eigene Sessions, aber Pre-Check
     vereinfacht sich).
2. **Server (`PartnerClientWelcomeBlock`)** rendert `<IchWillMehrCard>`
   nur wenn Prop gesetzt — kein Flash der Card bei "bereits gepushed".
3. **Client (`IchWillMehrCard`)** verwaltet Modal-Open-State via `useState`,
   uebergibt `captureSessionId` an `<IchWillMehrModal>`.
4. **Client (`IchWillMehrModal`) Submit**:
   - `e.preventDefault()` blockiert Default-Form-Submit
   - Defense-in-Depth: `!consentChecked` → setzt Error-Label, returns frueh
   - `startTransition` wrappt Server-Action-Call (Pending-State + Doppel-Submit-Schutz)
5. **Server-Action (`requestLeadPush`)** (MT-5 already QA'd):
   - Re-Validiert `consent_checkbox_value` (lead-push-actions.ts:89)
   - Re-Validiert `consent_text_version` (lead-push-actions.ts:96)
   - Inserted `lead_push_consent` + `lead_push_audit` + ggf. `ai_jobs`
   - Returns `{ ok, auditId } | { ok: false, error }`
6. **Modal-Result-Handling**:
   - `ok` → Success-State + `router.refresh()` (Server-Branch erkennt
     beim naechsten Render: Consent existiert → Card weg)
   - `already_pushed` (Race-Condition zweiter Tab) → wird als Erfolg
     dargestellt + Refresh entfernt Card
   - Andere Errors → ERROR_LABEL-Map oder Generic-Fallback

**Keine Wiring-Gaps detektiert.** Alle Glieder der Kette sind explizit
verbunden, keine Stub-Endpoints.
