## Problems found

1. **Slice-Spec-Drift: Storage-Pfad-Annahme.** Die Slice-Spec spricht von "alle .md-Files unter snapshot.storage_path" — real ist `storage_path` aber EIN ZIP-File `{tenant_id}/{snapshot_id}.zip`. Loesung: Reader entpackt das ZIP server-seitig per jszip im Memory. Kein Worker-Eingriff noetig. Bundle-Auswirkung: 0 auf Browser (jszip nur in Server-Components).
2. **Section→Block-Key-Mapping nicht eindeutig.** Sections in `template.handbook_schema` koennen `block_keys: []` (mehrdeutig) oder `block_keys: ['A']` (eindeutig) haben. Loesung: Cross-Link nur fuer Sections mit genau einem `block_keys`-Eintrag. Pragmatisch fuer V4.1 — V5 koennte einen Dropdown bieten.
3. **Pre-existing TS-Errors in `bridge/__tests__/action-helpers.test.ts:127,129`** (aus SLC-035, dokumentiert in RPT-101 + RPT-102). Build ist nicht betroffen, aber `tsc --noEmit` zeigt sie. **NICHT in diesem Slice gefixt** (nicht im Scope) — bleibt Open Point fuer SLC-043 oder ein V4.2 BL-Item.
4. **Bridge-Konsolidierungs-Frage offen.** Aus Memory-Handoff: "Werden Mitarbeiter-KUs via Bridge-Engine in die GF-Session konsolidiert, oder hat der Worker-KU-Loader denselben Filter-Drift wie ISSUE-029?" Antwort: Der Worker-KU-Loader filtert ueber `capture_session_id = snapshot.capture_session_id` (GF-Session). Das ist designmaessig korrekt — die Bridge-Engine spawnet Mitarbeiter-Sessions als separate Capture-Sessions, deren Output fliesst NICHT direkt ins Handbuch. Mitarbeiter-Inhalte landen via `block_review` (mit `source = 'employee_questionnaire'`) im Handbuch — und der Worker filtert diese KUs in `block-review-filter.ts` korrekt. Im Reader-Smoke wird das beim ersten Snapshot-Render direkt sichtbar (Mitarbeiter-Sektionen mit Inhalt = via Bridge konsolidiert; Mitarbeiter-Sektionen leer = Worker-Bug, der dann gefixt werden muesste). Strukturell aktuell stimmig.
