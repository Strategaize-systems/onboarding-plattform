## Findings

### Low

#### L-6 (NEU) — Resolver-Default-DisplayName "Strategaize" leakt im RPC-Fehler-Edge-Case in den Welcome-Block

**Schwere**: Low (UX-Edge-Case, kein Security-Issue, kein Datenverlust, nur sichtbar bei tatsaechlichen RPC-Fehlern).

**Befund**: `dashboard/page.tsx:81`
```typescript
let partnerDisplayName: string | null = branding.displayName;
if (!partnerDisplayName && tenantRow.parent_partner_tenant_id) {
  // fallback to partner_organization.display_name
}
```

Wenn die RPC `rpc_get_branding_for_tenant` fehlschlaegt (Try/Catch in `resolve.ts:69-71`), liefert der Resolver `STRATEGAIZE_DEFAULT_BRANDING` mit `displayName: "Strategaize"`. Die `if (!partnerDisplayName ...)`-Bedingung triggert nicht (truthy String), das Fallback auf `partner_organization.display_name` wird uebersprungen, und der Welcome-Block zeigt "Ihr Steuerberater: Strategaize" — verwirrend fuer den Mandanten.

**Trigger-Wahrscheinlichkeit**: Sehr selten. RPC + Postgres im selben Docker-Netzwerk, kein externes Hop. Nur bei Container-Crash oder DB-Outage.

**Empfohlener Fix** (in MT-12 oder Gesamt-/qa SLC-104):
```typescript
import {
  resolveBrandingForTenant,
  STRATEGAIZE_DEFAULT_BRANDING,
} from "@/lib/branding/resolve";

// ...
let partnerDisplayName: string | null =
  branding.displayName &&
  branding.displayName !== STRATEGAIZE_DEFAULT_BRANDING.displayName
    ? branding.displayName
    : null;
if (!partnerDisplayName && tenantRow.parent_partner_tenant_id) {
  // existing fallback
}
```

Alternative: Resolver liefert separates `isFallback: boolean` Flag — wuerde aber API-Breaking-Change in `BrandingConfig`-Interface sein.

#### L-7 (NEU) — Resolver wird 2x pro Mandanten-Page-Load aufgerufen

**Schwere**: Low (Performance, nicht Korrektheit).

**Befund**: Pro `/dashboard`-Request bei Mandanten:
1. `src/app/layout.tsx:21` ruft `resolveBrandingForCurrentRequest()` → `resolveBrandingForTenant(supabase, tenantId)`.
2. `src/app/dashboard/page.tsx:71-74` ruft erneut `resolveBrandingForTenant(supabase, profile.tenant_id)`.

Ergebnis: 2 identische RPC-Calls + 2 identische `rpc_get_branding_for_tenant`-Auswertungen pro Page-Load. Bei Mandanten-Skala (~10-100 Mandanten pro Partner, ~3-10 Page-Loads/Tag pro Mandant) noch tragbar, aber unnoetig.

**Empfohlener Fix** (Slice-Spec selbst nennt `ggf.` Optional in Section B):
- React `cache()` um `resolveBrandingForTenant` in `resolve.ts`. Deduplikation auf Request-Scope, kein cross-Request-Cache.
- Slice deferred es bewusst — V6.1+ Optimierung.

**Workaround derzeit**: keiner noetig. RPC ist <5ms.
