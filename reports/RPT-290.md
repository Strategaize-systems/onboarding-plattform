## Open Risks

### R-1 (Low) — Live-Smoke nach Coolify-Redeploy ausstehend
Migration 096 ist LIVE in DB, aber app + worker Container laufen noch auf Image-Tag `c3e9539` (V6.3 Code ohne neue Lookup-Logik). **Inkonsistenz im laufenden System:**
- DB-Constraint: UNIQUE(slug, version) ✅
- App-Code: `.eq("version", "v1").single()` ❌ (alter Code)

**Symptom waehrend dieser Inkonsistenz:** Die alte App-Code-Version mit `.eq("version", "v1").single()` funktioniert weiter (klappt deterministisch fuer die einzige existierende v1-Row), weil die Lookup-Aenderung backwards-kompatibel ist. Risiko nur bei einer zukuenftigen `partner_diagnostic_v2`-Migration ohne vorher den Code zu redeployen — dann wuerde die alte App weiterhin nur v1 finden statt v2. **Aktueller Stand 0 v2-Rows in DB.**

**Mitigation:** `/deploy V6.4` als REL-019 + User-Pflicht Coolify-Redeploy ASAP. Nach Redeploy: AC-11+AC-12 Live-Smoke (existing V6.3-Test-Session-Bericht + neue Diagnose-Session via Playwright MCP).

Existierende capture_session-Rows in der DB referenzieren via `template_id` weiter ihre Original-Template-Row (jetzt `partner_diagnostic v1`). Nach einer hypothetischen `partner_diagnostic v2`-Anlage:
- Alte Sessions: zeigen weiter auf v1 → bericht/page.tsx laedt v1 → originale Texte sichtbar.
