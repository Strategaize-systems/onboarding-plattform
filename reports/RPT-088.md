## Findings

### F4 — SOP-Renderer-Schema-Mismatch (HIGH)

**Severity:** High
**Area:** Renderer / Output-Quality
**AC:** AC-4 (Markdown syntaktisch valide), AC-3 (Section-Files vollstaendig), AC-10 (Counts vs. Inhalt — sop_count zaehlt korrekt, Inhalt aber unvollstaendig)

**Beobachtung:**
Der SOP-Renderer in `src/workers/handbook/sections.ts:339-348` erwartet
`step.title` und `step.detail`. Die echte SOP-Datenstruktur (definiert in
`src/workers/sop/types.ts` und vom SOP-Generator-Worker geschrieben) hat aber:

```typescript
interface SopStep {
  number: number;
  action: string;
  responsible: string;
  timeframe: string;
  success_criterion: string;
  dependencies: string[];
}
```

Dadurch ist `step.title` immer `undefined` und der Renderer faellt auf
`Schritt ${i}` zurueck. `step.detail` ist auch immer `undefined` -> kein
Detail-Text.

**Output-Beispiel (Live, Section 01):**

```markdown
### SOP: Geschäftsmodell & Markt - Transparenz und Bewertbarkeit herstellen

_Ziel:_ Vollständige Transparenz über Geschäftsmodell ...

1. **Schritt 1**
2. **Schritt 2**
3. **Schritt 3**
4. **Schritt 4**
5. **Schritt 5**
6. **Schritt 6**
7. **Schritt 7**
```

Erwartet (basierend auf DB-Daten):

```markdown
1. **Management-Interview zur Geschäftsmodell-Dokumentation durchführen**
   _Verantwortlich:_ Geschäftsführung + externer Berater | _Frist:_ 48 Stunden
   _Erfolg:_ Vollständiger Business Model Canvas mit Kerngeschäft, Zielkunden und Value Propositions dokumentiert
```

**Root-Cause:**
Die Test-Fixture in `src/workers/handbook/__tests__/fixtures.ts:SOP_BLOCK_A`
erfindet ein nicht-existentes Format `{title, detail}`. Die 5 renderer.test.ts
Tests rendern korrekt aus dieser Fixture und sind gruen — testen aber das
falsche Format. Der Bug ist deshalb durch Phase 1 nicht entdeckt worden.

**Fix:**
1. `sections.ts` `renderSop()`: SopStep aus `sop/types.ts` importieren, Titel
   aus `step.action` ableiten, Detail aus `step.responsible + step.timeframe + step.success_criterion`
   konstruieren (dependencies optional als Liste).
2. `fixtures.ts` SOP_BLOCK_A Step-Format auf echtes SopStep-Schema aendern.
3. renderer.test.ts Tests entsprechend nachziehen.

**Empfehlung:** Auto-Fix in dieser /qa-Session direkt nach User-Bestaetigung,
oder als Mini-Patch-Slice vor /backend SLC-040. NICHT in SLC-040 mischen
(SLC-040 ist UI; F4 ist Worker-Output).

### F6 — Signed-URL via Public-URL erfordert apikey-Query (Medium)

**Severity:** Medium
**Area:** Self-Hosted-Supabase + Coolify-Routing
**AC:** AC-7 (Signed-URL-Roundtrip)

**Beobachtung:**
Die Server-seitig erzeugte Signed-URL zeigt auf den internen Kong-Endpoint
(`http://supabase-kong:8000/storage/v1/object/sign/...?token=...`). Wenn die
URL ohne Anpassung ans Frontend uebergeben wird, schlaegt der Download via
Public-URL fehl mit HTTP 401 `{"message":"No API key found in request"}`.

Workaround: An die Signed-URL `&apikey=<NEXT_PUBLIC_SUPABASE_ANON_KEY>`
anhaengen und die Host-Replacement von `supabase-kong:8000` auf den
Public-Origin (`https://onboarding.strategaizetransition.com/supabase`) machen.

**Ueberpruefung im Test:**

```bash
# Mit apikey-Query: HTTP 200, 4524 bytes valider ZIP
curl "https://...supabase/storage/v1/object/sign/handbook/...?token=...&apikey=<ANON>"

# Ohne apikey-Query: HTTP 401
curl "https://...supabase/storage/v1/object/sign/handbook/...?token=..."
```

**Implikation fuer SLC-040:**
Die Server-Action `getHandbookDownloadUrl` muss:
1. RPC `rpc_get_handbook_snapshot_path` aufrufen -> liefert storage_path
2. `adminClient.storage.from('handbook').createSignedUrl(path, 300)` aufrufen
3. **Den Host der zurueckgelieferten signedURL** von `http://supabase-kong:8000`
   auf `process.env.NEXT_PUBLIC_SUPABASE_URL` ersetzen
4. **`apikey` Query-Param** mit `NEXT_PUBLIC_SUPABASE_ANON_KEY` anhaengen
5. Erst die finale URL an das Frontend zurueckliefern

Das matcht IMP-166 (Self-Hosted Public-Storage 3 verzahnte Fallen).

**Empfehlung:** Als Slice-Spec-Note in SLC-040 hinzufuegen, dort im /backend
korrekt umsetzen, im /qa SLC-040 explizit testen.

### F3 — Test-Daten haben keine confirmed Diagnose (Low)

**Severity:** Low (Test-Daten-Limitation)
**Area:** QA-Test-Fixtures
**AC:** AC-10

**Beobachtung:**
Die einzige Diagnose in der Test-Session 64ad04eb hat status='draft'.
Section 01 Schema-Filter hat `min_status: 'confirmed'`. Damit wird die
Diagnose deterministisch herausgefiltert -> diagnosis_count=0 im Snapshot.

Das ist erwartetes Verhalten und konsistent. Aber: Der `subsections_by='subtopic'`
Render-Pfad (sections.ts:184-245) wird damit Live nicht ausgeloest, weil keine
Subtopics zur Verfuegung stehen -> Fallback auf `subsections_by='block_key'`.

**Empfehlung:** Bei naechstem QA-Pass eine confirmed Diagnose erzeugen
(via SQL UPDATE block_diagnosis SET status='confirmed' WHERE id=...) und den
subtopic-Render-Pfad live verifizieren. Phase-1-Unit-Tests decken den Pfad
ab — Live-Verifikation ist Bonus.
