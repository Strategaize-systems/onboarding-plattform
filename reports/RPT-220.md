### 4. Compensating-Action-Pattern Verifikation (AC #2)

actions.ts:140-152 (`createPartnerOrganization`):

```ts
// Phase 2 — Bei partner_organization-INSERT-Fehler: Compensating-Delete des Tenants
if (poErr) {
  const { error: deleteErr } = await admin
    .from("tenants")
    .delete()
    .eq("id", tenantRow.id);
  if (deleteErr) {
    captureException(new Error(deleteErr.message), {
      source: "admin/partners/createPartnerOrganization/compensatingDelete",
      ...
    });
  }
  captureException(new Error(poErr.message), { ... });
  return { ok: false, error: "partner_organization_insert_failed" };
}
```

Pattern: Supabase JS-Client unterstuetzt keine BEGIN/COMMIT — daher Compensating-Delete des `tenants`-Row bei `partner_organization`-INSERT-Fehler. Worst-Case bei Compensating-Failure: Orphan-Tenant ohne `partner_organization`-Eintrag, der von `/admin/partners`-Liste (LEFT JOIN) als "Kein Owner / Kein partner_organization" angezeigt wuerde. Slice-Spec akzeptiert das, weil:

- `tenants`-INSERT-Fehler-Wahrscheinlichkeit sehr niedrig
- `partner_organization`-INSERT-Fehler nach `tenants`-INSERT sehr niedrig (gleiche Connection, gleiche TX-Latenz-Fenster)
- Compensating-Delete-Fehler tendenziell nur bei Connection-Loss → Orphan-Tenant wird durch Manual-Cleanup beseitigt

Vitest in `partners-actions-db.test.ts` deckt den Pfad ab (RPT-218 erwaehnt 4 V6-MT-1-Tests inkl. atomic rollback).
