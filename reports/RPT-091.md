## Live-Cross-Tenant-Test (Detailprotokoll)

```sql
-- Setup: zweiter Tenant + tenant_admin in BEGIN/ROLLBACK (replica-Mode umgeht handle_new_user-Trigger)
BEGIN;
SET LOCAL session_replication_role = replica;
INSERT INTO tenants (id, name) VALUES ('00000000-0000-0000-0000-0000000000bb', 'QA-Test-Foreign-Tenant');
INSERT INTO public.profiles (id, tenant_id, role, email)
  VALUES ('00000000-0000-0000-0000-000000000bb1',
          '00000000-0000-0000-0000-0000000000bb',
          'tenant_admin', 'qa-foreign@test');

SET LOCAL session_replication_role = origin;
SET LOCAL ROLE authenticated;
SET LOCAL request.jwt.claims TO '{"sub":"00000000-0000-0000-0000-000000000bb1","role":"authenticated"}';

-- Identity-Check
SELECT auth.uid(), auth.user_role(), auth.user_tenant_id();
-- Result: 0...0bb1 | tenant_admin | 0...0bb

-- Cross-Tenant-Versuch: Snapshot 86d18dd6 gehoert zu Tenant 0...0de
SELECT rpc_get_handbook_snapshot_path('86d18dd6-a73d-424f-9a65-a01b573b3f8e'::uuid);
-- Result: {"error": "forbidden"}                                    ✓

-- Direct-Table-Access unter RLS
SELECT count(*) FROM handbook_snapshot WHERE id = '86d18dd6-...';
-- Result: 0                                                         ✓

ROLLBACK;
```

Bemerkung: `auth.user_role()` und `auth.user_tenant_id()` lesen aus `public.profiles WHERE id = auth.uid()`, NICHT aus JWT-Claims. Damit ist ein Cross-Tenant-Bypass per gefaelschtem JWT-Claim auf `tenant_id` zusaetzlich blockiert — Identity wird immer aus der DB gelesen.
