- Beim ersten `git pull origin main` auf `/opt/onboarding-plattform-test` (Hetzner 159.69.207.29) verweigert Git den Merge wegen lokaler Aenderungen an `src/lib/db/__tests__/v6-partner-rls.test.ts` (+489 LOC).
- Diagnose: `git diff origin/main -- <file>` lieferte LEEREN Diff — die "lokalen Aenderungen" waren bereits identisch zu origin/main (commit 7a84d4d aus MT-11). Vermutlich CRLF-Whitespace-Drift oder eine ungespeicherte Server-side-Iteration aus MT-11-Debug, die nicht commited wurde.
- Fix: `git stash push -m 'MT-11-stale-2026-05-13'` + `git pull origin main` + `git stash drop 0`. Kein Datenverlust, Vitest danach gruen.
- Lehre: Pflicht-Pre-Pull-Check `git status --short` + `git diff origin/main` vor stash/reset bewahrt vor versehentlichem Loeschen echter In-Progress-Arbeit.

**Container-Suffix-Drift seit MT-11:**
- Coolify-Postgres-Container-Name war zu MT-11-Zeitpunkt (reference_coolify_test_setup.md, 2026-05-11) `supabase-db-bwkg80w04wgccos48gcws8cs-083715570318`. In dieser Session (2026-05-13) ist es `supabase-db-bwkg80w04wgccos48gcws8cs-073209612941`. Coolify-Project-ID `bwkg80w04wgccos48gcws8cs` und Netzwerk-Name `bwkg80w04wgccos48gcws8cs_strategaize-net` bleiben stabil; nur der Timestamp-Suffix aendert sich bei Redeploy.
- Lehre: Container-Suffix nicht hardcoden in Doku; immer per `docker ps --format ... | grep` aufloesen. Memory `reference_coolify_test_setup.md` braucht keinen Update — die Anweisung dort sagt schon "Container-Namen haben Coolify-Resource-UUIDs als Suffix — bei Repo-Re-Deploy aendert sich der Suffix nicht, der bleibt stabil pro Coolify-Resource". Das war eine Fehlannahme — der Suffix aendert sich bei jedem Container-Restart, NICHT pro Coolify-Resource. **Diese Drift sollte als IMP im Dev-System erfasst werden.**
