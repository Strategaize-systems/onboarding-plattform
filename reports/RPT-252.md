## Phase 5 — Production-Incident BS Postgres + Recovery

**Unbeabsichtigter Test-Treiber**: Beim ersten HTTP-Push (Phase 3, attempt=1)
schlug der synchrone Adapter-Call gegen `business.strategaizetransition.com`
mit HTTP 500 fehl, weil Business-System Postgres im Crash-Loop hing:

```
2026-05-14 10:25:31.624 UTC [7623] PANIC:  could not write to file
  "pg_logical/replorigin_checkpoint.tmp": No space left on device
2026-05-14 10:25:31.741 UTC [1] LOG:  checkpointer process (PID 7623)
  was terminated by signal 6: Aborted
```

Disk-Usage auf 91.98.20.191: `/dev/sda1 38G  37G  0 Avail (100%)`.
Hauptverbraucher: Docker Build Cache 4.32GB + unbenutzte Images 5.55GB
(zusammen ~10GB reclaimable). Postgres-Daten nur 64MB (vernachlaessigbar).

**Recovery-Aktion** (User-genehmigt via AskUserQuestion):
```bash
docker builder prune -af   # reclaim 5.59GB
docker image prune -af     # reclaim 2.45GB
```
→ Disk 100% → 83% (6.2GB free), Postgres `pg_is_in_recovery()=f` ✓, DB online.

**Das war ein realer Production-Incident im Business-System**, aber als
Nebeneffekt hat es die Retry-Pfad-Verifikation (MT-13) ermoeglicht — ein
echter "BS-Downtime-Smoke" statt eines simulierten Test-Cases.
