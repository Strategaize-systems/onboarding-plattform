## 3. Findings

### Blocker

- **Migration 065 `handle_new_user()` ohne `public.`-Prefix** — Finding aufgedeckt in-QA, sofort gefixt.
  Root cause: `CREATE OR REPLACE FUNCTION handle_new_user()` ohne Schema-Prefix. Postgres-search_path in Supabase-Selfhost ist `"", storage, public, extensions` — die Funktion landete in `storage.handle_new_user` statt `public.handle_new_user` zu ueberschreiben. Der Trigger `on_auth_user_created` auf `auth.users` ruft explizit `public.handle_new_user()` auf, also greift weiterhin die alte Whitelist ohne `'employee'`. Folge: Fixtures-Setup in `src/__tests__/rls/v4-fixtures.ts` schlug mit `handle_new_user: invalid role: employee` fehl, ALLE 11 V4-Matrix-Tests fielen aus.
  Fix: Migration 065 erweitert um `CREATE OR REPLACE FUNCTION public.handle_new_user()` (explizit) + `DROP FUNCTION IF EXISTS storage.handle_new_user() CASCADE` (Cleanup). Idempotent. Commit `d83c9a9`, pushed. Verifikation: erneuter Test-Lauf liefert 45 Tests (11 passed + 34 todo, 0 failed).
  IMP-120 im Dev-System angelegt.

_(keine)_
