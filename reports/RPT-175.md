## Most Likely Cause / Narrowed Cause Area

### Cause 1 — Sidebar
Trivial: SLC-075 hat die Routen + Self-Spawn-Action implementiert, aber `src/components/employee-sidebar.tsx` nicht erweitert. Slice-Spec hatte das Sidebar-Update nicht explizit als MT aufgefuehrt — eine echte Implementation-Luecke des `/backend`-Schritts.

### Cause 2 — Upload-Fehler (Hairpin-NAT auf Self-Hosted Supabase)

Diagnose nach Layer-Pattern:

**Layer 1 (Container health):** PASS — `app-bwkg80w04...` Up healthy mit Image-Tag `20eef2db21cd...`.

**Layer 2 (ENV im app-Container):**
```
NEXT_PUBLIC_APP_URL=https://onboarding.strategaizetransition.com
NEXT_PUBLIC_SUPABASE_URL=https://onboarding.strategaizetransition.com/supabase   ← extern
SUPABASE_URL=http://supabase-kong:8000                                            ← intern
```

**Layer 4 (Application logic):** `src/lib/supabase/admin.ts` Zeile 20 nutzt `process.env.SUPABASE_URL || process.env.NEXT_PUBLIC_SUPABASE_URL!` — picked also den internen Hostnamen. Wenn `createSignedUploadUrl` aufgerufen wird, baut supabase-js die `signedUrl` als String relativ zur Client-Base — Resultat ist `http://supabase-kong:8000/storage/v1/object/upload/sign/walkthroughs/...?token=...`. Diese URL wurde an den Browser zurueckgegeben.

Browser → DNS-Lookup von `supabase-kong` schlaegt fehl (intern Docker-Network only) → XHR triggert `onerror` → User-sichtbarer Fehler. Coolify-app-Logs sind clean (PUT erreicht den Server nie).

**Vergleich mit existierendem Code:** Evidence-Upload (`src/app/api/capture/[sessionId]/evidence/upload/route.ts`) nutzt das **API-Route-Proxy-Pattern** (Browser → Next.js multipart/form-data → Server-side Upload via interner Supabase-URL), nicht das Direct-PUT-Pattern. SLC-071 hat erstmals im Onboarding-Repo den Browser-Direct-PUT-Pattern eingefuehrt — und damit erstmals die Hairpin-NAT-Falle getroffen.

### Fix 2 — signedUrl-Rewrite Helper

```ts
// src/app/actions/walkthrough.ts
function rewriteSignedUrlForBrowser(url: string): string {
  const internalBase = process.env.SUPABASE_URL;
  const externalBase = process.env.NEXT_PUBLIC_SUPABASE_URL;
  if (!internalBase || !externalBase) return url;
  if (internalBase === externalBase) return url;
  if (!url.startsWith(internalBase)) return url;
  return externalBase + url.slice(internalBase.length);
}

// In requestWalkthroughUpload-Return:
return {
  walkthroughSessionId: session.id,
  uploadUrl: rewriteSignedUrlForBrowser(signed.signedUrl),
  storagePath,
};
```

Effekt fuer Coolify-Pfad:
- Vor Rewrite: `http://supabase-kong:8000/storage/v1/object/upload/sign/walkthroughs/...?token=...`
- Nach Rewrite: `https://onboarding.strategaizetransition.com/supabase/storage/v1/object/upload/sign/walkthroughs/...?token=...`

Coolify-Reverse-Proxy routet `/supabase/*` → `supabase-kong:8000/*`, also erreicht der PUT die Storage-API extern korrekt. Pattern fuer Supabase-Cloud (intern==extern): kein Rewrite, URL bleibt unveraendert.

## Residual Uncertainty

- **Storage-Bucket-RLS / Public-Endpoint**: Ich habe nicht geprueft, ob die `walkthroughs`-Bucket-Policy die Coolify-Public-Route richtig handhabt. Wenn die Bucket-Policy `apikey`-Header verlangt, kann es trotz Rewrite zu 401/403 kommen. ISSUE-025 (V4) hatte ein verwandtes Phaenomen ("Signed-URL via Public-Endpoint erfordert apikey-Query-Param"). Falls AC-12 erneut fehlschlaegt mit anderem Status (z.B. HTTP 401/403): Bucket-Policy + Coolify-Reverse-Proxy-Forwarding pruefen.
- **Multipart-Upload bei sehr grossen Files**: `createSignedUploadUrl` mit `upsert:false` macht Single-PUT. Bei 30min-Recording + viel Content kann das nahe der Storage-Default-Limits sein (50MB-100MB typisch). Wenn AC-12 mit grossen Files fehlschlaegt (HTTP 413): pre-flight File-Size-Check + Bucket-Limit erhoehen.
- **CORS auf der Public-Route**: Storage-API muss `Access-Control-Allow-Origin: <app-domain>` zurueckgeben. Falls Coolify-Reverse-Proxy CORS strippt: Browser-Fehler im DevTools-Console pruefen. Bei AC-12-FAIL Round 2: User soll die Browser-DevTools-Network-Tab oeffnen + den failed PUT klicken — Status-Code + Headers zeigen das tatsaechliche Verhalten.
