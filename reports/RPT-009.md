## Deviation-Report

- **Rule 1 (Auto-fix):** Im ersten Test-Run schlug der WITH-CHECK-Test fehl, weil PostgreSQL nach einem RLS-INSERT-Error die Tx in Abort-State setzt und das `finally`-`RESET ROLE` fehlschlug. Fix mit SAVEPOINT/ROLLBACK-TO-SAVEPOINT angewandt. **Angewandt.**
- **Rule 1 (Auto-fix):** Erste Fixture + Tests wurden gegen falsche Schema-Felder geschrieben (z.B. `payload` statt `content`, `fact` statt `observation`, `'draft'` statt `'open'`). Aus Migration 021 nachgezogen. **Angewandt.**
- **Rule 2 (Auto-add missing):** Fixture um Template erweitert (capture_session braucht template_id NOT NULL). Ohne Template keine Sessions anlegbar → blocker-fix. **Angewandt.**
- **Rule 3 (Auto-fix blocking):** Auf dem Hetzner-Server fehlte Node + npm. Installiert via `curl nodesource/setup_20.x | bash -` + `apt-get install nodejs`. **Angewandt** (notwendig fuer Tests).
- **Rule 3 (Auto-fix blocking):** Erster Test-Run mit `node:20-alpine` schlug fehl wegen musl/glibc-Inkompatibilitaet zu node_modules (Host ist glibc). Wechsel auf `node:20` (Debian-basiert). **Angewandt.**
- **Rule 4 (Stop & ask):** Keine architektonische Aenderung. Tests laufen serverseitig im Coolify-Netzwerk, nicht lokal auf Windows — das ist der pragmatische V1-Weg, nicht ein neuer Tech-Stack.

## Findings

### Medium

**M1: Test-Run-Setup fuer lokale Entwicklung nicht optimiert**
- **Fund:** Die Tests laufen im Docker-Container im Coolify-Netzwerk auf Hetzner. Lokal auf Windows braeuchte der User einen SSH-Tunnel oder separaten Postgres. Beides ist dokumentiert, aber setup-Reibung bleibt.
- **Impact:** Dev-Cycle-Zeit haengt vom SSH-Tunnel ab. Bei haeufigen Test-Runs spuerbar langsamer als lokaler Postgres.
- **Empfehlung:** Follow-up-Task (kein Blocker): docker-compose.test.yml mit lokalem postgres-Container + Init-Scripts automatisiert laden. Wird erst relevant wenn Test-Volumen steigt (ab SLC-005+).

**M2: `package.json.name` ist `ai-coding-starter-kit` (Blueprint-Erbe)**
- **Fund:** Scope von ISSUE-005 / SLC-002c. Nicht Teil von SLC-002a.
- **Empfehlung:** Wird in SLC-002c bereinigt.

### Low

**L1: Test-Coverage noch nicht on-demand verifiziert**
- **Fund:** Acceptance #3 verlangt "lesbaren Coverage-Report". Script ist konfiguriert, aber erster Run auf Hetzner nicht durchgefuehrt (unnoetig fuer diesen Slice, der nur 1 Test-Datei enthaelt).
- **Empfehlung:** Bei erster umfangreichen Testsuite (SLC-005+) einmal `npm run test:coverage` ausfuehren und HTML-Report anschauen.

**L2: Tests in /opt/onboarding-test parallel zu Coolify-Checkout**
- **Fund:** Der Test-Checkout auf Hetzner ist ein separater Clone unter `/opt/onboarding-test`, nicht der Coolify-Arbeitsordner. Das ist bewusst — Coolify-Checkout nicht mit npm-install-for-tests verschmutzen.
- **Empfehlung:** Akzeptiert. Clone-Pfad in README dokumentieren, falls User es selbst nachfahren will.

## Open Risks

- **Lokal-Dev-Setup auf Windows:** Funktioniert nur mit SSH-Tunnel oder Docker-Linux-VM (kein native psql-Wrapper). Bis jemand lokal Tests intensiv laeuft, kein Blocker.
- **Coverage-Thresholds:** Nicht erzwungen. Erst wenn Baseline-Coverage bekannt ist, sollte man Minimum-Werte festlegen (SLC-005+).
