## Problems Found

### P-1 ‚Äî Coolify-Postgres search_path: `postgres`-User hat `storage` VOR `public`

**Severity:** Medium (in MT-4a beim ersten Apply konkret gefunden, sofort gefixt).

**Befund:** Beim ersten Apply von Migration 094 wurde `CREATE OR REPLACE FUNCTION rpc_finalize_partner_diagnostic(...)` ohne Schema-Prefix ausgefuehrt. `\df rpc_finalize_partner_diagnostic` zeigte die Function im `storage`-Schema. Ursache: `SHOW search_path` fuer `postgres`-User auf der Coolify-DB liefert `"", storage, public, extensions` ‚Äî `storage` (Supabase-Bucket-Schema) steht VOR `public`, sodass schemafreie DDL dort landet.

**Behebung:** Migration 094 auf `public.rpc_finalize_partner_diagnostic` explizit umbenannt (CREATE FUNCTION + COMMENT + GRANT). Cleanup: `DROP FUNCTION IF EXISTS storage.rpc_finalize_partner_diagnostic(jsonb);` zusaetzlich im Migrations-File (idempotent, raeumt die fehlplatzierte Function aus dem ersten Apply auf). Zweiter Apply 18:11 UTC: `CREATE FUNCTION / COMMENT / GRANT / DROP FUNCTION` alles erfolgreich, Verifikation zeigt Function nur noch in `public`-Schema.

**Folge:** **Empfehlung fuer alle zukuenftigen `rpc_*`-Migrationen auf Coolify-Onboarding-Plattform-DB: IMMER `public.<function_name>` explizit als Prefix verwenden.** Bestehende RPCs (z.B. `rpc_bulk_import_knowledge_units` aus MIG-035) sind nicht betroffen, weil sie aelter sind und vermutlich vor dem Setup des aktuellen search_paths angelegt wurden ‚Äî Verifikation via `\df rpc_*` zeigt sie alle im `public`-Schema.

### P-2 ‚Äî `light-pipeline-score.test.ts` Logger-Eager-Init-Regression durch MT-4-Import

**Severity:** Low (sofort durch `vi.mock` im Test-File geheilt; kein Production-Code-Touch noetig).

**Befund:** MT-4 hat in `light-pipeline.ts` den Import `import { captureException } from "../../lib/logger";` hinzugefuegt. Das macht den MT-3-Pure-Function-Test `light-pipeline-score.test.ts` bei Modul-Load instabil: `lib/logger.ts` initialisiert auf Top-Level einen `createClient(...)`, der ohne `SUPABASE_URL`-ENV mit `validateSupabaseUrl: supabaseUrl is required` crasht. Vor MT-4 hat der Score-Test diese Eager-Init nicht angetriggered, weil keine logger-Reference im Pure-Function-Path lag.

**Behebung:** `vi.mock("../../../lib/logger", () => ({ captureException: vi.fn(), captureInfo: vi.fn() }));` als Top-Level-Statement in den Score-Test eingefuegt. 18/18 Tests bleiben gruen ohne Production-Code-Aenderung.

**Folge:** Pattern fuer alle Tests, die `light-pipeline.ts`-Module importieren: `vi.mock("../../../lib/logger", ...)` ist Pflicht. Dasselbe gilt fuer andere Module die `lib/logger` importieren (siehe `src/workers/lead-push/__tests__/handle-job.test.ts` ‚Äî pre-existing failed, NICHT durch MT-4 verursacht).
