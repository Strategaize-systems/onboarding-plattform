# BL-076 — capture-reminders Cross-Day-Idempotenz-Hotfix

## Outcome Summary

ISSUE-035 (Stage1-Mehrfach-Send ueber Tage) ist code-side resolved. Atomare Aenderung in 3 Files, 17/17 Vitest PASS, TypeScript clean. Live-Verifikation erfolgt erst nach User-Coolify-Deploy + naechstem Cron-Run (Trigger `0 9 * * *` Europe/Berlin).

## Root Cause

`pickStage(acceptedAt, now)` in [src/lib/reminders/process-reminders.ts](../src/lib/reminders/process-reminders.ts) entscheidet allein anhand der **aktuellen** Werktage-Differenz:

```ts
if (workdays >= 14) return null;
if (workdays >= 7)  return "stage2";
if (workdays >= 3)  return "stage1";
return null;
```

→ An Werktag 3, 4, 5, 6 wird `"stage1"` zurueckgegeben. Cron laeuft taeglich → Stage1 wird taeglich neu ausgeloest.

Die UNIQUE-Constraint `reminder_log_employee_user_id_reminder_stage_sent_date_key` blockiert nur Same-Day-Duplikate, nicht Stage1-Mehrfach-Send ueber 4 Tage. Beobachtung in Live-DB (RPT-162): an `richard@bellaerts.de` 4× Stage1 (01-04.05.) + 1× Stage2 (05.05.).

## Fix

**Cross-day idempotency** ueber Status-Lookup im `reminder_log`:

1. **Type-Erweiterung** in [process-reminders.ts](../src/lib/reminders/process-reminders.ts):
   ```ts
   export interface ReminderCandidate {
     ...
     // Stages mit status='sent' fuer diesen User (egal welches Datum).
     already_sent_stages?: ReminderStage[];
   }
   ```

2. **Guard-Clause** in `processReminders` vor `insertLog`/`sendMail`:
   ```ts
   if ((c.already_sent_stages ?? []).includes(stage)) {
     result.skipped_already_sent++;
     continue;
   }
   ```

3. **Loader** in [route.ts](../src/app/api/cron/capture-reminders/route.ts) `loadCandidates`:
   ```ts
   const { data: sentLogs } = await supabase
     .from("reminder_log")
     .select("employee_user_id, reminder_stage")
     .in("employee_user_id", userIds)
     .eq("status", "sent");
   // → Map<user_id, Set<stage>> → already_sent_stages pro Candidate
   ```

**Wichtige Design-Entscheidungen:**

- **Nur status='sent' blockiert.** `status='failed'` blockiert NICHT — wenn SMTP an Tag 3 gefailt ist, soll Tag 4 retry. `status='skipped_opt_out'` blockiert auch nicht — wenn User Opt-out deaktiviert, soll naechster Run zustellen.
- **Optionales Feld + defensive Lese-Logik (`?? []`).** Vermeidet Refactoring-Aufwand in 4 existierenden Test-Candidates und macht zukuenftige Caller-Migration weicher.
- **Kein DB-Constraint-Aenderung.** Die UNIQUE-Constraint `(user, stage, sent_date)` bleibt — sie schuetzt weiterhin gegen Mehrfach-Send pro Tag (z.B. Cron-Doppellauf). Cross-Day-Schutz wird auf Application-Level gehoben, weil das logisch flexibler ist (Status-Filter, Retry-Pfade).
