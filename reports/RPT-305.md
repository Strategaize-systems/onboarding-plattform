## Pen-Test Findings (Hauptzweck der Suite)

### F-1 (BLOCKER / RESOLVED) — Migration 098 fehlt service_role + authenticated GRANTs auf `pending_signup`

**Symptom:** Pen-Test Case 3 (validKey + neue Email → 202) failt mit `error: { code: '42501', message: 'permission denied for table pending_signup' }`. POST /api/public/signup wuerde in Production beim ersten echten Signup-Versuch einen 500-Error mit `internal_error` werfen — captureException schreibt `[object Object]` als message, was die Diagnose erschwert.

**Root Cause:** Migration 098 (SLC-132) hat `pending_signup` mit `ENABLE ROW LEVEL SECURITY` angelegt, aber das uebliche Supabase-Pattern `GRANT ALL ON public.<table> TO service_role, authenticated` vergessen. Zum Vergleich: `profiles`-Tabelle hat fuer service_role 7 Privilegien (INSERT/SELECT/UPDATE/DELETE/TRUNCATE/REFERENCES/TRIGGER), `pending_signup` hatte ZUVOR NUR postgres-Owner-Grants. RLS-Policies sind keine GRANTs — sie filtern Rows nach Auth-Context aber RLS-Pruefung passiert NACH dem GRANT-Check. Ohne GRANT bekommt jeder Aufruf bereits am ersten Pruefpunkt `permission denied`.

**Auswirkung:** **Production-Outage-Garantie** beim ersten echten Public-Signup. Pre-Production-Live-Smoke (in SLC-135 vorgesehen) haette das sofort entdeckt — aber dann waere ein Hotfix unter Zeitdruck noetig gewesen. Pen-Test SLC-134 hat den Bug zwei SLCs frueher entdeckt, mit Zeit fuer saubere Reproducibility + Cause-Diagnose + Migration-Fix.

**Hotfix (Deviation Rule 1):**
1. `sql/migrations/098a_v7_pending_signup_grants.sql` erstellt: `GRANT SELECT,INSERT,UPDATE,DELETE ON public.pending_signup TO service_role; GRANT ... TO authenticated;`. Idempotent.
2. Auf Coolify-DB `supabase-db-bwkg80w04wgccos48gcws8cs-102228495360` LIVE per base64+psql-U-postgres ~16:58 UTC.
3. **PostgREST schema-Cache reload PFLICHT:** `NOTIFY pgrst, 'reload schema'` — sonst zeigt der `/rest/v1/pending_signup`-Endpoint HTTP 404 statt 201 (PostgREST cached Schema-Metadata, neue GRANTs werden erst nach Reload aktiv).
4. Re-Test: MT-2 Pen-Test 12/12 PASS gegen geupdate-DB.

**SLC-135 Implication:** Im Live-Smoke-Step muss nach jeder neuen Migration ein `NOTIFY pgrst, 'reload schema'` automatisch ausgefuehrt werden. Sonst entsteht ein 5-Minuten-404-Fenster nach jedem Deploy.

**MIG-043** in `/docs/MIGRATIONS.md` dokumentiert die Hotfix-Migration mit voller Rollback-Notes-Sektion.

### F-2 (Medium) — supabase-js Error-Object-Maskierung erschwert Production-Diagnose

**Beobachtung:** Wenn der ECHTE PostgreSQL-Fehler `permission denied for table pending_signup` ist, gibt `supabase-js .insert(...).select(...).single()` ein leeres Error-Object `{}` zurueck. Der Production-Code (`route.ts:378`) ruft dann `captureException(e, ...)`, was am Ende ein error_log-Eintrag mit message `[object Object]` erzeugt — die eigentliche PostgreSQL-Fehlermeldung geht verloren.

**Auswirkung:** Production-Debugging bei DB-Fehlern ist mit aktuellem error_log-Output schwierig — Operations sieht nur `[object Object]` und muss raten, welche Tabelle/Action betroffen war.

**Empfohlene Massnahme (NICHT V7-Scope, V7.1 Polish):** Logger-Wrapper fuer supabase-js-Errors. captureException sollte bei `e` mit shape `{ code, message, details, hint }` (PostgrestError) den eigentlichen `message` + `code` als string serialisieren.

**Status:** Dokumentiert, kein Hotfix in SLC-134 (out-of-scope).

### F-3 (Low) — Pen-Tests sind flaky im Volltree-Parallel-Run

**Beobachtung:** Pen-Test-Files isoliert (5 Files, 25 Tests) laufen 25/25 PASS in 1.49s. Im Volltree-Lauf (111 Files, 1156 Tests) failen sporadisch 2 SLC-134-Tests:
1. **MT-5 Timing-Test**: Mean-Time-Differenz bei 4 Varianten ueberschreitet 1us-Bound (Beobachtung: 2-3us Mean unter Volltree-CPU-Load).
2. **MT-4 Case 18 (Race)**: parallele Tests legen ebenfalls partner_client_mapping-Rows an, die im JOIN-Query mitgezahlt werden → `expected 0 to be 1` Counter-Mismatch.

**Root Cause:** Vitest Default-Mode laeuft Tests parallel ueber Worker. Timing-Test misst CPU-Zeit (verzerrt bei Worker-Last). Race-Test zaehlt DB-Rows (kontaminiert bei parallel-DB-Inserts in selbem Test-Run).

**Empfohlene Massnahme (V7.1 Polish, NICHT V7-Scope):**
1. `vitest.config.ts` um `poolMatchGlob` erweitern: Pen-Test-Files in eigenem Fork-Pool mit `singleFork: true`.
2. Oder: Separates `npm run test:pen-test`-Skript fuer Slice-Schluss-/qa, das nur die 5 Pen-Test-Files seriell laeuft.

**Status:** Im RPT dokumentiert. Volltree-CI-Fail wird in /qa SLC-134 als "Known-Flaky, isolated PASS" eingeordnet (Slice R-1 + R-2 Mitigation aktiv).
