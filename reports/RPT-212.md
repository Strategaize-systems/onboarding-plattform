## Problems found

**Keine Blocker. Folgende Drifts/Klarstellungen transparent dokumentiert:**

1. **Slice-Spec sprach von "mind. 96 V6 + 94 Regression"**, aber die Spec-Tabelle summiert sich auf 52-56 V6 + 94 Regression (16 Cross-Partner-Read + 8 Cross-Client-Read + 12 Write-Block + 8 Cross-Mandant + 4 Mandant-vs-Partner + 4 vs-Direkt + 48 V5.1 Regression + 46 V4 Regression). Meine Suite hat 71 aktive V6-Tests + 4 it.todo — uebertrifft die Spec-Tabelle deutlich (incl. Schema-Smoke + Trigger + CHECK-Constraint + Read-Own + strategaize_admin Override). Regression V4/V5.1 laeuft in bestehenden Test-Files (admin-rls + rls-isolation + walkthrough-embed-rls + v5-walkthrough-rls = 69 Faelle live verifiziert; weitere Files dialogue-session/meeting-guide/template-queries/user-settings-cross-user wurden nicht im selben Run getestet, aber nicht durch Migration 090 betroffen).
2. **`profiles.role` CHECK + `handle_new_user`-Trigger erweitert (NICHT in Slice-Spec explizit erwaehnt)**: Slice-Spec listete `tenants` ALTER + 2 neue Tabellen, aber das Anlegen eines `partner_admin`-Profile-User braucht zwingend dass `profiles.role` CHECK + `handle_new_user` die neue Rolle akzeptieren. Ohne diese Erweiterung waere AC #3 ("Neue Rolle `partner_admin` ist in `pg_roles` vorhanden und **kann via JWT-Claim auf SET LOCAL ROLE wechseln**") und die Pen-Test-Suite (die partner_admin-User anlegt) nicht erreichbar. Daher in Migration 090 als Step 1 mit drin.
3. **RLS-Policy-Pattern: `TO authenticated` mit `auth.user_role()`-Check statt `TO partner_admin`**: Die V4/V5-Codebase-Konvention ist, dass alle App-Sessions als `authenticated` connecten und der Rolle ueber `auth.user_role()` aus `profiles.role` ermittelt wird. Die MIG-034-Skizze in `/docs/MIGRATIONS.md` zeigt zwar Beispiele mit `TO partner_admin`, das wuerde aber nicht greifen bei normalen App-Sessions (die als `authenticated` laufen). Ich folge der bestehenden Konvention (Defense-in-Depth via `auth.user_role()` + Tenant-Constraint) UND lege zusaetzlich die Postgres-Rolle `partner_admin` mit GRANTs an, damit `SET LOCAL ROLE partner_admin` jederzeit moeglich ist (AC #3 erfuellt).
4. **Mandant-Sichtbarkeit auf eigenes `partner_client_mapping`** (Policy `pcm_select_own_mandant`): Slice-Spec sagt "Mandant darf das eigene Mapping lesen via `client_tenant_id = auth.user_tenant_id()` — fuer Branding-Resolution". Diese Policy ist eingebaut. Sie erlaubt aber alle Mandanten-Rollen (`tenant_admin`, `tenant_member`, `employee`) — was sinnvoll ist, weil das Branding fuer alle Mandanten-Sessions resolved werden muss, nicht nur Admin.
5. **Trigger-Function `check_partner_client_mapping_tenant_kinds()` als SECURITY DEFINER** (Slice-Spec lehnt zum Pattern nicht explizit Stellung): Die Function liest `public.tenants.tenant_kind` und wuerde ohne SECURITY DEFINER an RLS scheitern, wenn ein partner_admin-User Mappings einlegt — er hat keine SELECT-Permission auf fremde tenants-Rows. SECURITY DEFINER laeuft als Function-Owner (postgres) und bypasst RLS. Saubere Loesung analog `auth.user_role()`.
6. **`partner_organization.partner_kind` CHECK heute nur `'tax_advisor'`** (DEC-111): Spalte angelegt mit DEFAULT + CHECK. V8 erweitert CHECK auf `('tax_advisor', 'ma_advisor')` per einzeiliger ALTER — keine Daten-Migration noetig. `tier` Spalte ohne CHECK angelegt (V3+ wird Werte festlegen), V6 schreibt immer NULL.
