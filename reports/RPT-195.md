## Rollback Plan

**RTO ~5min Coolify-Redeploy. RPO 0 Datenverlust.**

### Trigger fuer Rollback

- SC-V5-1 oder SC-V5-4 schlagen mit Hard-Crash fehl
- Externe Routen liefern persistent 5xx > 10min nach Deploy
- Worker-Pipeline-Stages bleiben > 2h in `redacting`/`extracting`/`mapping` ohne Fortschritt (Cleanup-Cron faengt das nach 1h, aber wenn Cron selbst broken: manueller Eingriff)
- error_log-Spam mit V5-induzierten Errors > 50 in der ersten Stunde

### Rollback-Schritte

1. **Code-Rollback:** Coolify UI → Onboarding-App → Deploy-Commit `9be520b` (V4.4 REL-012 Stand) → Redeploy
2. **DB-Rollback:** **NICHT erforderlich.** Migrations 082..087 sind alle additiv:
   - 082: capture_session.capture_mode CHECK ext (additiv)
   - 083: walkthrough_session CREATE TABLE
   - 084: walkthroughs Storage-Bucket CREATE (idempotent)
   - 085: walkthrough_step CREATE TABLE
   - 086: walkthrough_review_mapping CREATE TABLE
   - 087: walkthrough_session.status + knowledge_unit.source CHECK-Erweiterung (mit DROP IF EXISTS-Pattern, idempotent)

   V4.4-Code ignoriert die V5-Tabellen + neuen Status-Werte. **Stale-Sessions in `redacting/extracting/mapping`** bleiben bis Cleanup-Cron sie nach 1h auf `failed` setzt — danach faellt sie nach 7d in den failed-Storage-Delete-Pfad.

3. **Coolify-Cron-Cleanup:** Falls V4.4-Code den walkthrough-cleanup-cron-Endpoint nicht hat (404), Scheduled-Task in coolify-db.scheduled_tasks `enabled=false` setzen ODER auf hoehere Frequenz reduzieren bis Re-Deploy.
4. **Storage-Cleanup:** `walkthroughs`-Bucket bleibt — keine Daten loeschen (Recovery-Quelle).
