# /doctor SLC-075 Round 2 — Kong apikey-Query-Param Injection

## Problem Summary

Nach Round-1-Hotfix (Commit `bf2b146`) und User-Browser-Smoke Round 2:
- **AC-10 PASS**: Sidebar-Eintrag "Walkthroughs" sichtbar (ISSUE-037 Round 1 funktioniert)
- **AC-11 PASS**: Self-Spawn → Permissions → MediaRecorder startet
- **AC-12 FAIL Round 2**: Upload schlaegt jetzt mit `HTTP 401 {"message":"No API key found in request"}` fehl — anderer Bug als Round 1 (`xhr.onerror` Netzwerkfehler).

## Observed Symptoms

User-Screenshot (Round 2):
- URL: `/employee/walkthroughs/75098a5d.../record`
- Inline-Fehlerbox: `Fehler: Upload-Fehler: Upload fehlgeschlagen mit HTTP 401 { "message":"No API key found in request" }`

Diff zu Round 1: HTTP 401 statt Network-Error. Heisst: PUT erreicht den Server (Coolify-Reverse-Proxy → Kong), aber Kong rejects mangels apikey-Header/Query.

## Most Likely Cause

Round-1-Fix hat den Host umgeschrieben: `http://supabase-kong:8000/...` → `https://onboarding.strategaizetransition.com/supabase/...`. Coolify-Reverse-Proxy routet `/supabase/*` → Kong. Kong erwartet als API-Gateway den `apikey`-Header oder Query-Param fuer JEDEN Request — auch fuer signed-URL-Requests, die self-contained Auth via Token haben.

`createSignedUploadUrl` baut die signed URL mit `?token=<jwt>` aber OHNE `apikey`-Query. supabase-js setzt den `apikey`-Header normalerweise auto fuer Requests via Client. Bei direkter Browser-PUT bekommt Kong den Header nicht, der Token-only-Request wird mit 401 rejected.

V4 ISSUE-025 (Status: wontfix, Resolution Date: 2026-04-27) hat genau dasselbe Symptom dokumentiert: *"Signed-URL via Public-Endpoint erfordert apikey-Query-Param"*. V4 hat das durch API-Route-Proxy umgangen (Browser → Next.js → Server-side download). Fuer Walkthrough-Upload (potentiell 200MB+) ist API-Route-Proxy ineffizient (doppelte Bandbreite); Direct-PUT mit apikey-Query-Param ist die kanonische Self-Hosted-Loesung.

## Recommended Fix

Implementiert. Helper-Logik nach Round 2:

```ts
function rewriteSignedUrlForBrowser(url: string): string {
  const internalBase = process.env.SUPABASE_URL;
  const externalBase = process.env.NEXT_PUBLIC_SUPABASE_URL;
  const anonKey = process.env.NEXT_PUBLIC_SUPABASE_ANON_KEY;
  if (!internalBase || !externalBase) return url;

  // Round 1: Host-Rewrite intern → extern (nur self-hosted)
  let rewritten = url;
  if (internalBase !== externalBase && url.startsWith(internalBase)) {
    rewritten = externalBase + url.slice(internalBase.length);
  }

  // Round 2: apikey-Query-Param fuer Coolify-Kong (ISSUE-025-Pattern)
  if (anonKey && !/[?&]apikey=/.test(rewritten)) {
    const separator = rewritten.includes("?") ? "&" : "?";
    rewritten = `${rewritten}${separator}apikey=${encodeURIComponent(anonKey)}`;
  }

  return rewritten;
}
```

Effekt fuer Coolify-Pfad nach Round 2:
- vor Round 1: `http://supabase-kong:8000/storage/v1/.../sign/...?token=t`
- nach Round 1: `https://onboarding.strategaizetransition.com/supabase/storage/v1/.../sign/...?token=t`
- nach Round 2: `https://onboarding.strategaizetransition.com/supabase/storage/v1/.../sign/...?token=t&apikey=<anon>`

Effekt fuer Cloud-Pfad:
- Host bleibt unveraendert (intern==extern)
- apikey wird trotzdem appended (harmlos, Cloud akzeptiert es gracefully)
- Falls signedUrl schon `apikey=` enthaelt: kein Doppel-Append (Idempotenz)

Alle drei sind antizipierbar, aber nicht durch dieses /doctor abgedeckt.
