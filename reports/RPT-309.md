- **Impact:** V7-Self-Signup-Funnel ist Cross-System-blocked. Der Mandant kann nirgendwo eine IS-Landing-Page sehen oder einen Verify-Flow anstossen. OP-Code ist bereit fuer Caller-Hits, aber kein Caller existiert.

#### F-2 — `src/lib/supabase/middleware.ts` updateSession-Whitelist fehlt `/api/public/*` + `/auth/verify-signup` → ALL Live-Endpoints 307 zu /login

- **Beobachtung:** Direkter Live-Test `curl -X POST https://onboarding.strategaizetransition.com/api/public/signup -H "x-strategaize-service-key: <KEY>" -d <BODY>` → HTTP 307 mit `Location: /login`. Identisch bei `/api/public/partner/qa-steuerberater-demo` (GET) und `/auth/verify-signup` (GET). Reproduziert AUCH von intern via `docker exec node -e "fetch('http://localhost:3000/api/public/signup', { redirect: 'manual' })"` → 307 → /login. Beweist: kein Traefik/Coolify-Issue, sondern Application-Layer-Proxy-Redirect.
- **Root Cause Code:** [`src/lib/supabase/middleware.ts:47-58`](src/lib/supabase/middleware.ts#L47-L58) `updateSession()` Whitelist enthaelt `isPublicPath` (statische `/login`, `/auth/callback`, `/auth/set-password`, `/accept-invitation`, `/datenschutz`, `/impressum`), `isApiHealth`, `isApiCron`, `isApiUnsubscribe`, `isApiPartnerBranding`. **Fehlt komplett: `isApiPublic` (Pattern `/api/public/*`) + `isAuthVerifySignup` (Pattern `/auth/verify-signup`).** Wenn kein `user`-Session und keine Whitelist matched, fuehrt der Fall `if (!user && !isPublicPath && !isApiHealth && !isApiCron && !isApiUnsubscribe && !isApiPartnerBranding)` zu `return NextResponse.redirect("/login")`.
- **Impact:** 100% der V7-Cross-System-Funnel-HTTP-Calls werden geblockt — V7-Verify-Endpoint kann nicht geklickt werden, IS-Caller-Calls werden umgelenkt, jeder Self-Signup-Browser-Hit endet auf der Login-Page. **Vollstaendiger Production-Outage des V7-Features**, sobald F-1 (IS-Caller) gefixt ist.
- **Warum von Slice-/qa nicht entdeckt:** Alle SLC-131..135 Vitest-Tests rufen Route-Handler-Functions DIREKT auf (Import + Function-Call), nicht via HTTP-Request-Pipeline. Der `updateSession()`-Proxy laeuft nur fuer echte HTTP-Requests durch Next-16 Proxy. Diese Test-Strategie ist als Mock-Drift-frei (Tests gegen reale Coolify-DB) korrekt, aber sie ignoriert die HTTP-Layer-Realitaet komplett.
- **Beweis dass es nie funktioniert hat:** error_log historische Eintraege vom 2026-05-19 10:04 zeigen 4 erfolgreiche `accepted_pending_verify`-Audit-Eintraege fuer signup-Route — das waren Vitest-Pen-Test-Runs gegen Coolify-DB, NICHT Live-HTTP-Requests. Heute (post-Redeploy) wurde der erste Live-HTTP-Request gemacht → 307-redirect-Pattern bestaetigt.
- **Fix:** 4-Zeilen-Source-Edit in `src/lib/supabase/middleware.ts`:
  ```typescript
  // FUEGE HINZU vor isApiCron-Zeile:
  const isApiPublic = pathname.startsWith("/api/public/");
  const isAuthVerifySignup = pathname.startsWith("/auth/verify-signup");
  
  // ERWEITERE die !user-Check-Zeile:
  if (!user && !isPublicPath && !isApiHealth && !isApiCron && !isApiUnsubscribe && !isApiPartnerBranding && !isApiPublic && !isAuthVerifySignup) {
  ```
  + Code-Review + Vitest-Test fuer Whitelist-Logic + Re-Live-Smoke. Aufwand ~15min Code + 15min Verifikation.
- **Wertung:** **BLOCKER fuer V7-Release.** Source-Code-Bug in OP, nicht in IS. Fix MUSS in V7.0-Hotfix oder direkt im V7-Deploy sein, sonst ist V7 unbenutzbar.
