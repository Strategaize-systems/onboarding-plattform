# V6 SLC-104 MT-9 — Welcome-Block Update mit Partner-Logo + Display-Name

## Outcome summary

**Status: complete (code-side).** MT-9 implementiert wie im Slice (Section H) spezifiziert. Mandanten-Welcome-Block erhaelt jetzt Partner-Branding ueber den vorhandenen Resolver — keine neue DB-Logik, kein neuer Endpoint, reine Wiring-Arbeit.

9/13 MTs in SLC-104 done. Offene MTs: MT-10 RPC-Vitest gegen Coolify-DB, MT-11 Pen-Test-Faelle aktivieren, MT-12 Quality-Gates Gesamt, MT-13 Browser-Smoke nach Coolify-Deploy.

## Implementation

### PartnerClientWelcomeBlock (Server Component)

- Props erweitert um `partnerDisplayName?: string | null` und `partnerLogoUrl?: string | null` — beide optional, damit der Block bei Resolver-Fehlern (R-104-1) auf den generischen SLC-103-Hinweis zurueckfaellt.
- Logo-Block: nur wenn `partnerLogoUrl` ODER `partnerDisplayName` gesetzt. `next/image` mit `unoptimized` (Server-Proxy-URL, kein Build-time-Optimize), 48x48 Container mit `object-contain`, Alt-Text aus Display-Name oder neutralem "Partner-Logo".
- Begruessungstext: "Empfohlen von {partnerLabel}." mit `partnerLabel = partnerDisplayName?.trim() || "Ihrem Steuerberater"`. Trim verhindert leere-String-Edge-Cases aus DB.
- Inline `<p>Powered by Strategaize</p>` (vorher line 71) entfernt — duplikative Inhalts-Zeile gegenueber dem globalen `StrategaizePoweredFooter` aus MT-4/5 im Root-Layout. Tightly-scoped Cleanup, weil die Datei sowieso fuer MT-9-Branding-Wiring angepasst wurde.
- Ungenutzten `Link from "next/link"`-Import (war seit SLC-103 dead) durch `Image from "next/image"` ersetzt.

### dashboard/page.tsx (Partner-Client-Branch)

- Import `resolveBrandingForTenant` aus `@/lib/branding/resolve` ergaenzt.
- Bestehender Admin-Client-Lookup auf `tenants` zusaetzlich `parent_partner_tenant_id` selektiert (vorher: nur `name, tenant_kind`).
- Im `tenant_kind === 'partner_client'`-Branch: `resolveBrandingForTenant(supabase, profile.tenant_id)` — die SECURITY-DEFINER RPC traversiert intern zum parent (Slice Section A.2), `branding.logoUrl` ist die Server-Proxy-URL `/api/partner-branding/<mandant-id>/logo` (Proxy ruft denselben RPC auf, faellt also ebenfalls auf parent zurueck).
- Display-Name-Fallback-Kette (kein neuer Helper, einfacher Inline-Code):
  1. `branding.displayName` (aus `partner_branding_config.display_name`, optional NULL by default)
  2. `partner_organization.display_name` (NOT NULL, Pflichtfeld seit Migration 090) — Lookup nur wenn (1) leer + `parent_partner_tenant_id` vorhanden, ueber Admin-Client (Mandant hat keinen direkten SELECT auf `partner_organization`, RLS-Boundary).
  3. `null` -> Welcome-Block faellt auf "Ihrem Steuerberater" zurueck.
- Direkt-Kunden (`tenant_kind=direct_client`) bleiben **regression-frei** — der ganze neue Code lebt im bestehenden partner_client-Branch.

### Auswirkung auf den Render-Pfad

- Browser-Cache des Logos: Server-Proxy setzt `Cache-Control: public, max-age=3600` (siehe `route.ts:122`). Branding-Aenderungen werden also nach max 1h sichtbar — Mandant kann mit Hard-Reload sofort den neuen Stand sehen. Akzeptiertes V6-Tradeoff (siehe MT-7 Comment).
- CSS-Custom-Properties (Akzentfarbe) waren bereits in MT-5 (Root-Layout) live — MT-9 ergaenzt **nur** Display-Name + Logo im Begruessungs-Bereich.
