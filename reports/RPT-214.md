## Problems found

**Keine Blocker. Eine Deviation transparent dokumentiert:**

1. **Compensating-Action statt DB-Tx (R-102-1 Mitigation)**: Slice MT-1 verlangt "BEGIN TX" fuer atomare 3-INSERT-Sequenz (tenants + partner_organization + error_log/audit). Supabase/PostgREST kann ueber HTTP keine Multi-Statement-Transaction ohne eigene SECURITY-DEFINER-RPC ausdruecken. Migration 090 enthaelt keine solche RPC. Statt einer neuen Migration in MT-1 (waere ein 091a-Eintrag mit Renumbering-Risiko) verwende ich ein **2-Phasen-INSERT mit explizitem DELETE-Cleanup** bei Phase-2-Failure (`partner_organization` INSERT scheitert → vorheriger `tenants` INSERT wird per service_role wieder geloescht). AC #2 ("kein Orphan-Tenant") ist funktional erfuellt. Im Vitest-Test wird die Compensating-Action durch eine UNIQUE-Violation auf `partner_organization.tenant_id` provoziert und das DELETE auf `tenants` verifiziert (4. Test-Case in partners-actions-db.test.ts). Dokumentiert in Code-Header der actions.ts. **Folgeueberlegung**: Falls Production-Live-Daten zeigen, dass Partial-Failure-Pfade haeufig sind, sollte in V6.1 eine dedizierte SECURITY-DEFINER-RPC `rpc_create_partner_organization` als Backlog-Item entstehen (analog `rpc_create_employee_invitation`).

2. **error_log.category-Spalte existiert nicht** (Slice MT-1 referenziert sie): Tabelle `error_log` (Migration 039) hat Spalten `level/source/message/stack/metadata/user_id/created_at`. **Loesung**: `category` als JSONB-Key in `metadata.category` umgesetzt (`metadata: { category: "partner_organization_created", ... }`). Konsistent mit dem captureInfo+captureException-Pattern aus `src/lib/logger.ts`.

3. **invitePartnerAdmin nutzt employee_invitation-Tabelle (Schema-Reuse statt Schema-Erweiterung)**: Slice MT-1 erlaubt "INSERT `invitation` (oder bestehendes Schema)" — die existierende `employee_invitation`-Tabelle hat ein freitext-`role_hint`-Feld. Ich schreibe `role_hint='partner_admin'`. Der MT-6-Accept-Pfad muss spaeter `role_hint` lesen und beim `auth.admin.createUser` entsprechend `user_metadata.role` setzen (statt hardcoded `"employee"`). Keine Schema-Migration noetig. Dokumentiert in actions.ts.

4. **Live-Tests gegen Coolify-DB als postgres-Superuser, nicht service_role**: Die DB-Tests in `partners-actions-db.test.ts` laufen via `withTestDb` als postgres-Superuser. Die produktiven Server Actions nutzen service_role (`createAdminClient`). Beide bypassen RLS, beide arbeiten auf demselben Schema/Constraints — fuer die hier getesteten Faelle (Schema-Constraints, UNIQUE-Violations, Compensating Action) ist die Semantik identisch. Action-Layer-Validation (Email-Regex, Country-Enum, Auth-Check) ist pure Logik und wird in MT-3 ueber Form-Submit-Smoke abgedeckt — keine separate Vitest noetig.

5. **MT-Verify-Drift positiv**: Slice MT-2 Verify sagt "Vitest 4 Faelle: 4 Rollen × 2 Pfad-Klassen". Implementierte Suite hat 45 Vitest (6 Pfad-Klassen × 5 Rollen + classifyPath 18 + Login-Defaults 5). Uebertrifft Spec, schadet nicht.
