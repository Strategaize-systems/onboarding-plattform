## MT-6 Hetzner-Deploy — SQL-Block fuer User

Laut Rule `.claude/rules/sql-migration-hetzner.md`. Siehe `reference_hetzner_blueprint.md` (159.69.207.29) falls Zugang eines anderen Onboarding-Servers noch angelegt werden muss.

### Schritt 1 — Container-Namen ermitteln
```bash
docker ps --format "{{.Names}}" | grep -i supabase
```

### Schritt 2 — Migrationen per Base64 uebertragen

Migration 021:
```bash
# Base64 lokal generieren (vom Dev-System aus):
cd c:/strategaize/strategaize-onboarding-plattform
base64 -w 0 sql/migrations/021_core_capture_schema.sql
```

Auf dem Server (einzeiliger Befehl pro Migration):
```bash
echo '<BASE64_021>' | base64 -d > /tmp/021.sql
echo '<BASE64_022>' | base64 -d > /tmp/022.sql
echo '<BASE64_023>' | base64 -d > /tmp/023.sql
wc -l /tmp/021.sql /tmp/022.sql /tmp/023.sql
```

### Schritt 3 — Ausfuehren (als postgres, NICHT supabase_admin)
```bash
docker exec -i <supabase-db-container> psql -U postgres -d postgres < /tmp/021.sql
docker exec -i <supabase-db-container> psql -U postgres -d postgres < /tmp/022.sql
docker exec -i <supabase-db-container> psql -U postgres -d postgres < /tmp/023.sql
```

### Schritt 4 — Verifikation
```bash
docker exec -i <supabase-db-container> psql -U postgres -d postgres -c "\dt"
docker exec -i <supabase-db-container> psql -U postgres -d postgres -c "\d capture_session"
docker exec -i <supabase-db-container> psql -U postgres -d postgres -c "\d knowledge_unit"
docker exec -i <supabase-db-container> psql -U postgres -d postgres -c "SELECT policyname, tablename FROM pg_policies WHERE tablename IN ('template','capture_session','block_checkpoint','knowledge_unit','validation_layer') ORDER BY tablename, policyname;"
```

- Indizes laut Migration 023 vorhanden
