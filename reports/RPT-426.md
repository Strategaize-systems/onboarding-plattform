## Entscheidungs-Begruendung Kurz-Version

**AWS SES Inbound Ireland** gewinnt klar in 3 Dimensionen:

1. **Cost ~50-150x guenstiger** fuer V9.1-Volumen
   - 1k Mails/Mo: SES ~$0.11 vs Mailgun-Basic $15/Mo Plan-Mindestgebuehr
   - 10k Mails/Mo: SES ~$1.10 vs Mailgun-Foundation $35/Mo
   - 100k Mails/Mo: SES ~$11 vs Mailgun-Scale $90/Mo
   - Mailgun Dezember 2025 Pay-as-you-go-Preise verdoppelt → Trend teurer

2. **0 neue Vendor-Beziehung**
   - Bestehender AWS-Account (Bedrock-Frankfurt-Stack nutzt dasselbe AWS-Account)
   - AWS-Standard-DPA bereits aktiv, kein neuer Vertrags-Setup
   - Keine neue Sub-Processor-Liste pro [[data-residency]] Rule

3. **mailparser-Reuse aus V9.0**
   - Raw-Email-Persistierung in S3 + Worker-Read via mailparser = exakt das Pattern V9.0 SLC-165 MT-3
   - Mailgun-JSON-Payload waere paralleler Adapter-Stack der V9.0-MIME-Parsing-Infrastruktur dupliziert

## Trade-offs akzeptiert

1. **Region-Drift Ireland eu-west-1 statt Frankfurt eu-central-1**
   - AWS SES Inbound ist nicht in Frankfurt verfuegbar
   - Bedrock LLM bleibt Frankfurt
   - Cross-Region-Transfer innerhalb EU = DSGVO-konform, aber formell TIA-Punkt
   - In /architecture V9.1 als ADR-Audit-Punkt dokumentieren

2. **+2-4h Setup-Aufwand**
   - SES + S3 + SNS + Lambda + Endpoint + IAM gegenueber Mailgun-Direct-Webhook
   - Einmalig, danach Standard-AWS-Operations

3. **Mehr Moving Parts**
   - SES → S3 → SNS/Lambda → eigener Endpoint gegenueber Mailgun (Direct-HTTPS-Webhook)
   - AWS-IAM-Policy-Pflege als operative Last (aber bestehender Stack)

## Plan-B dokumentiert

**Mailgun EU** bleibt im PRD + DEC-194-Body als Plan-B fuer den Fall:
- /architecture V9.1 bewertet SES-Lambda-Pattern als unbillige Komplexitaet
- Founder bewertet Simplicity-Argumente staerker als Cost
- Region-Konsistenz Frankfurt eu-central-1 wird BLOCKING-Argument

Adapter-Pattern (Interface `InboundEmailVendor`) ermoeglicht Plan-B-Wechsel ohne Business-Logic-Refactor.

## Webhook-Architektur-Skizze (fuer /architecture V9.1)

```
Forward-Mail aus GF-Inbox
    ↓
MX-Record bulk.strategaizetransition.com
    ↓
AWS SES Inbound eu-west-1 (inbound-smtp.eu-west-1.amazonaws.com)
    ↓
SES Receipt-Rule-Set (Catchall *.bulk.strategaizetransition.com)
    ↓
S3-Bucket bulk-email-inbound
  Path: <tenant-slug>/<yyyy>/<mm>/<dd>/<message-id>.eml
    +
Lambda-Trigger "forward-ses-to-op-webhook"
    ↓
HMAC-signed POST → OP /api/inbound/email
  Header: X-AWS-SES-Signature (HMAC-SHA256 mit AWS_SES_INBOUND_SIGNING_SECRET)
  Body: { s3_path, tenant_slug, message_id, received_at, raw_size_bytes }
    ↓
OP-Endpoint:
  - HMAC-Verify
  - Tenant-Lookup via tenant_slug
  - Forward-Validation-Layer (FEAT-076)
  - email_message-INSERT mit storage_path=<s3_path>
  - ai_jobs-Enqueue email_bulk_parse fuer Worker
    ↓
Worker email_bulk_parse (V9.0 Reuse)
  - Storage-Read aus S3
  - mailparser-Parse (V9.0 Reuse parser.ts)
  - email_message-Felder fuellen
  - Trigger Pipeline-Stufen Pre-Filter → Threading → PII → Pattern → Curation
```

### P-1 — SES Inbound Region-Drift bleibt offener TIA-Punkt

DEC-194 nennt SES Receipt-Rule-Set + S3-Bucket-Erstellung + Lambda-Bridge + IAM-Role als /architecture-/backend-Aufgabe. Founder muss aber in AWS-Console (oder via IaC Terraform/CDK falls dort vorhanden) die initiale SES-Verifikation der `bulk.strategaizetransition.com`-Subdomain + MX-Record-Eintragung im DNS-Provider durchfuehren. Founder-Pflicht-Liste in /architecture V9.1 explizit erstellen.
