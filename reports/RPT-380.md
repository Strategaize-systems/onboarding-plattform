## Key Design Decisions (in MT-2a getroffen)

1. **`DO $mig051$ BEGIN ... END $mig051$`-Wrap** statt nur `BEGIN; ... COMMIT;` — fuer `RAISE NOTICE`-Logging in PL/pgSQL-Block. View + Bucket-INSERT ausserhalb des DO-Blocks, weil CREATE VIEW nicht in PL/pgSQL ohne EXECUTE laeuft.
2. **Status-Index als Partial-Index** — `WHERE status NOT IN ('completed', 'failed')` reduziert Index-Groesse auf aktive Runs (die meisten Rows sind langfristig completed/failed).
3. **MIME-Type-Allowlist `bulk-email`-Bucket** — `application/mbox`, `message/rfc822`, `application/octet-stream`, `text/plain`. `application/octet-stream` toleriert weil Browser bei `.mbox` oft so labelt; `text/plain` weil .eml-Files manchmal so kommen.
4. **knowledge_unit.source CHECK NICHT erweitert** — R5 aus Slice-Spec geprueft. Per ARCHITECTURE.md Line 8075-8091 (DEC, kein Migration-Bedarf) wird `metadata->>'source_type'='email_bulk'` als JSONB-Wert verwendet. R5 ist informational, nicht blockierend.
5. **Auth.users-INSERT-Pattern reused aus v4-fixtures.ts** — Supabase auth.users-Schema erfordert viele NOT-NULL Spalten (instance_id, aud, role, encrypted_password, raw_app_meta_data, raw_user_meta_data). Mein erster Versuch mit `INSERT INTO auth.users (id, email) VALUES (gen_random_uuid(), 'mt2-test@example.test')` haette gegen NOT-NULL gecrasht.
