## Rollback Plan

**Trigger-Bedingungen fuer Rollback:**
- Reader (`/dashboard/handbook/[snapshotId]`) wirft Crash-Loop bei Live-Snapshot-Last
- Berater-Workflow (`/admin/blocks/.../review`) bricht Cross-Tenant-Isolation
- Worker `handle-snapshot-job` bricht bei Pre-Filter (`loadApprovedBlockKeys` Exception)

**Rollback-Procedure (aus MIG-028 Rollback Notes):**

```bash
# 1. Coolify-UI: App-Container-Image zurueck auf V4-Tag (Pre-V4.1)
#    → Worker-Pre-Filter ist damit deaktiviert
#    → Reader-Routes 404 (akzeptabel, da V4 sie nicht hatte)

# 2. SSH zur DB: SQL-Rollback ausfuehren (nur falls Tabelle entfernt werden soll)
#    Achtung: Pre-V4.1 V4-Workflow funktioniert OHNE diesen Schritt weiter,
#    weil die Tabelle additiv ist. Schritt 2 nur bei explizitem Schema-Cleanup.
ssh root@159.69.207.29 \
  "docker exec -i supabase-db-bwkg80w04wgccos48gcws8cs-173534197991 \
   psql -U postgres -d postgres -c '
     DROP TRIGGER IF EXISTS tg_block_review_pending_on_employee_submit ON capture_event;
     DROP FUNCTION IF EXISTS tg_block_review_pending_on_employee_submit();
     DROP TABLE IF EXISTS public.block_review CASCADE;
   '"

# 3. Frontend-Revert ist mit Schritt 1 implizit erledigt (App-Image enthaelt UI).
```

**Recovery Time Objective (RTO):** ~5 Minuten (Coolify-Image-Rollback ist 1-Klick + Container-Restart).

**Recovery Point Objective (RPO):** 0 Datenverlust. Pre-V4.1-Daten sind in V4-kompatiblem Zustand. block_review-Eintraege wuerden bei Schema-Drop verloren gehen, sind aber rekonstruierbar (Backfill ist deterministisch).

**Was Pre-V4.1 weiterhin funktional bleibt:**
- alle V4-Snapshots werden weiter generiert (Worker-Pre-Filter ist Best-Effort, fehlende Eintraege = approved per DEC-048)
- alle V3-Capture-Flows
- alle V2-Templates
