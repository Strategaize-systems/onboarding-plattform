## Design Decisions (inline)

Folgende Entscheidungen wurden Simplicity-First gesetzt — keine neuen DECs:

- **DEC-MT-6-A Sequenzielle Thread-Verarbeitung** (nicht parallel): MT-6 verarbeitet Threads strikt sequenziell innerhalb des Workers. Vorteil: kein concurrent-Bedrock-Quota-Risk, kein Race auf email_thread INSERT, einfacher Status-Maschine-Walk. Tradeoff: bei N>30 Threads pro Run kann der Worker mehrere Minuten laufen. Per V9.0-Discovery typischer Bulk-Run hat <40 content-Emails → <10 Threads → akzeptabel.
- **DEC-MT-6-B `thread_status='redacting'` vor Bedrock-Call** (nicht erst nach): INSERT mit `thread_status='redacting'` schon vor dem Bedrock-Call macht den DB-State sichtbar. Bei Crash bleibt der Operator-Cleanup-Pfad klar. Final-Status 'redacted' via separate UPDATE nach Erfolg.
- **DEC-MT-6-C Idempotency via `existingRoots.has(root_message_id)`**: bestehende email_thread (gleicher bulk_run_id + root_message_id) werden vollstaendig geskipped (kein Re-Bedrock-Call, kein Pseudonyms-Refresh). Vorteil: deterministisch, kein Doppel-Charge. Tradeoff: wenn der erste Run Pseudonyms falsch belegt hat, wird der Re-Run das nicht reparieren — Operator muss manuelles DELETE WHERE thread_status='failed' + Re-Trigger machen. V9.0 akzeptiert per R3-Spec.
- **DEC-MT-6-D `thread_count = neu + bestehende`**: am Run-Ende wird `email_bulk_run.thread_count` mit der Summe gesetzt, nicht nur mit den neu persistierten. So sieht die UI nach Re-Run die Total-Anzahl, nicht nur die Delta-Anzahl. Pattern-Konsistent mit V9-Status-Semantik.
- **DEC-MT-6-E Single chatCaller pro Thread** (nicht pro Email): konsequent aus MT-5 DEC-MT-5-B. Concat-Body mit `=== Email N ===`-Separator (MT-5 buildEmailRedactUserMessage). V9.1+ koennte per-Email-Loop bei >30-Email-Threads.
- **DEC-MT-6-F ai_cost_ledger Non-Fatal**: Cost-Audit-Insert-Error wird ge-loggt aber wirft NICHT. Pipeline-Erfolg ist wichtiger als Cost-Tracking (V5-Pattern). MT-6 dokumentiert das ueber Tests Case 10.
- **DEC-MT-6-G tenantDomainResolver Default-Stub** (`undefined`): default-Resolver liefert `undefined` zurueck, sodass extractParticipantMap die first-seen-Reihenfolge nutzt. GF-Priority-Mapping ist V9.1+-Feature und braucht echten Tenant-Domain-Lookup aus `tenants.primary_domain` o.ae. Spec-Intent: MT-6 ist Worker-Core, GF-Priority kann in MT-7 oder V9.1+ nachgeruestet werden.
- **DEC-MT-6-H Pre-existing-Bug-Fix in Migration 108** (statt separate Migration): ai_jobs.job_type CHECK-Erweiterung waere strenggenommen Spec-orthogonal (Spec MT-6 nennt nur role-Extension). Aber: ohne diesen Fix waere MT-6-Trigger (MT-3 Approval-Button) live tot. Auto-Fix Rule 1 + 2 rechtfertigt Inlining in Migration 108 statt separate Migration 109. Atomic in einer Transaction.
