## Problems found

- **P-1 (search_path-Schema-Drift bei CREATE FUNCTION)**: Erster Apply von 092a
  schrieb `rpc_claim_next_ai_job_for_type` in `storage` statt `public`, weil der
  `postgres`-Superuser in self-hosted Supabase `search_path` = `"", storage,
  public, extensions` hat. Geist-Duplikat in storage, App rief weiter die alte
  public-Variante. **Reparatur**: Migration auf `CREATE OR REPLACE FUNCTION
  public.<name>(...)` + `DROP FUNCTION IF EXISTS storage.<name>(...)`
  umgestellt, zweiter Apply sauber. Verifikation per `SELECT nspname FROM
  pg_proc JOIN pg_namespace` zeigt nur `public`. **Dev-System-Lehre als
  IMP-507** in `strategaize-dev-system/docs/SKILL_IMPROVEMENTS.md` erfasst.

- **Spec-Drift Worker-Signatur**: Slice-Spec Section D zeigt Worker als
  `input: { job, adminClient }`-Objekt, das existierende JobHandler-Typsystem
  ist `(job: ClaimedJob) => Promise<void>`. Implementation hat sich an das
  bestehende Pattern gehalten (claim-loop dispatched mit Single-Arg). Slice-
  Spec-Note ist konzeptuell, kein Defekt.
