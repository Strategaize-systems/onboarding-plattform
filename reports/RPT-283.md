## Problems Found

### P-1 — Form-Action-Signature TS-Inkompatibilitaet (sofort gefixt)
**Severity:** Low (sofort beim ersten tsc-Run entdeckt + gefixt).

`<form action={serverAction}>` erfordert in TS-Strict-Mode eine Signatur `(formData: FormData) => void | Promise<void>`. Mein erster Entwurf von `startDiagnoseRun` hatte Signatur `(): Promise<{ error?: string }>` — tsc lehnt ab. Fix: Signatur auf `(_formData?: FormData): Promise<void>` umgestellt, Fehler via `throw new Error(...)` (next.js redirect throws ohnehin, also konsistent). `saveDiagnoseDraft` + `submitDiagnoseRun` behalten `{error}`-Returns weil sie via `useTransition` aus QuestionFlow gerufen werden, nicht via `<form action>`.

### P-2 — vi.mock-Hoisting-Falle (sofort gefixt)
**Severity:** Low (sofort gefixt).

Erste Testversion referenzierte `redirectMock` direkt im `vi.mock("next/navigation", () => ({ redirect: redirectMock }))` Factory. Vitest hoisted vi.mock vor die Module-Imports → `ReferenceError: Cannot access 'redirectMock' before initialization`. Fix: `vi.hoisted({ redirectMock: vi.fn(...), userClientMock: vi.fn(), adminClientMock: vi.fn() })` umstrukturiert. Pattern fuer alle zukuenftigen Tests mit Mocked-Module-References.

### P-3 — Answer-Key-Format-Drift (bewusst gewaehlt)
**Severity:** Low (Doku-Item, nicht Code-Item).

V6.3 partner_diagnostic-Template hat question.key bereits dotted (`ki_reife.q1`). Existing exit_readiness-Action schreibt `${blockKey}.${questionId}` (z.B. `A.<uuid>`). V6.3 schreibt direkt `answers[question.key]`. Vermischung waere kompatibel-brechend — daher separate Action-File `dashboard/diagnose/actions.ts` statt Reuse von `capture/[sessionId]/block/[blockKey]/actions.ts`. Dokumentiert im Header-Kommentar.
