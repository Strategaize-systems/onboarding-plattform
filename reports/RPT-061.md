## Acceptance Criteria Verification

### AC-1: Auftraggeber kann Dialogue-Session mit 2 Teilnehmern erstellen — PASS
- **Route**: `/admin/session/[sessionId]/dialogue/new` → Server Component laedt Members + Meeting Guide
- **Form**: 2 Dropdowns (Participant A vorselektiert auf current user, B gefiltert), Meeting-Guide-Info-Badge, Validierung (beide noetig, verschieden)
- **Wiring**: CreateDialogueForm → `createDialogueSession()` Server Action → `rpc_create_dialogue_session` RPC → `dialogue_session` INSERT → Redirect zu Meeting-Page
- **Error Handling**: RPC-Fehler angezeigt, UNIQUE-Constraint-Fehler behandelt
- Verifiziert: Code + TypeScript-Kompilierung

### AC-2: Consent-Screen vor Meeting-Beitritt (beide Teilnehmer) — PASS
- **Server-Logik**: `page.tsx:56-62` prueft `consent_a` / `consent_b` per Teilnehmer-Identitaet. strategaize_admin hat Bypass.
- **UI**: Fullscreen-Overlay mit Warnung-Icon, 2 Erklaerungstexte, Checkbox + Button. Button disabled bis Checkbox checked.
- **Wiring**: ConsentScreen → `updateDialogueConsent()` → `rpc_update_dialogue_consent` → setzt `consent_a` oder `consent_b` je nach User-ID
- **Blocking**: Ohne Consent kein Jitsi-IFrame (Consent-Screen blockiert Meeting-View)

### AC-3: Jitsi-Meeting laeuft im Browser (IFrame oder Link) — PASS
- **Script-Loading**: Dynamisches `<script>` von `https://{domain}/external_api.js`
- **IFrame API**: `new JitsiMeetExternalAPI(domain, { roomName, jwt, parentNode, config })` mit Branding-Overrides
- **Fallback**: Bei Script-Error → Direct-Link-Button (`https://{domain}/{room}?jwt={jwt}`)
- **JWT-Flow**: Consent → `generateDialogueJwt()` → Server Action prueft Participant-Identity → `generateJitsiJwt()` → JWT zurueck an Client
- **Config**: `prejoinPageEnabled: false`, `disableDeepLinking: true`, Watermarks deaktiviert
- Hinweis: Runtime-Verifikation erst nach Deploy moeglich

### AC-4: Meeting-Guide sichtbar waehrend des Meetings — PASS
- **Sidebar**: 80 Spalten breit (`w-80`), rechts neben Jitsi-IFrame via `children` Prop
- **Inhalte**: Sortierte Topics mit Nummerierung, Leitfragen als Bullet-Liste, Gesamtziel
- **Interaktion**: Checkbox "besprochen" pro Topic (gruener Hintergrund + Durchstreichung), Zaehler "{n}/{total} besprochen"
- **ScrollArea**: Bei vielen Topics scrollbar
- **Empty State**: "Keine Themen im Meeting-Guide" wenn leer
- **Guard**: Sidebar wird nur gerendert wenn `guideTopics.length > 0`

### AC-5: Recording-Status sichtbar (aktiv/inaktiv) — PASS
- **Event**: `recordingStatusChanged` von Jitsi API → `onRecordingStatusChange(active)`
- **UI**: RecordingIndicator zeigt rotes Pill-Badge mit Puls-Animation ("Aufnahme aktiv")
- **Status-Update**: Bei Recording-Start → `updateDialogueStatus(dialogueId, "recording")`
- **Conditional Render**: `if (!isRecording) return null` — zeigt nichts bei Inaktivitaet
- **Position**: Header-Bar rechts oben neben Status-Badge

### AC-6: Nach Meeting-Ende: Status wechselt, Pipeline wird getriggert — PASS
- **Event**: `videoConferenceLeft` → `handleMeetingEnd()` → `updateDialogueStatus(dialogueId, "completed")`
- **UI-Transition**: Status wechselt auf "completed" → Post-Meeting-Card mit "Meeting beendet" + "Wird verarbeitet..."
- **Redirect**: Nach 1.5s → `router.push()` + `router.refresh()` zurueck zur Meeting-Page (zeigt Post-Meeting-Status)
- **Pipeline**: Tatsaechlicher Trigger erfolgt via Recording-Webhook (SLC-028/030), nicht direkt aus Frontend. Frontend setzt Voraussetzung (status=completed).
