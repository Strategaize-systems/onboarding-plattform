## Findings (gruppiert nach Severity)

### High

#### F-110-H1 — ISSUE-049 cache()-Wrap dedupliziert in Production NICHT zwischen Layout und Page (supabase-Instanzen unterscheiden sich)

**Severity:** High (false-positive "done" — Fix loest die dokumentierte Symptom-Beschreibung nicht).

**Wiring-Trace:**

```
src/app/layout.tsx
  → resolveBrandingForCurrentRequest()  [resolve-server.ts:16]
    → const supabase = await createClient()  [resolve-server.ts:18]   <-- supabase-Instanz #1
    → resolveBrandingForTenant(supabase, tenantId)  [resolve-server.ts:34]
      → cache().get([supabase#1, tenantId]) → MISS → RPC call #1

src/app/dashboard/page.tsx
  → const supabase = await createClient()  [page.tsx:26]                <-- supabase-Instanz #2
  → const branding = await resolveBrandingForTenant(supabase, profile.tenant_id)  [page.tsx:71-74]
    → cache().get([supabase#2, tenantId]) → MISS (Object.is(#1,#2)===false) → RPC call #2
```

`createClient()` aus `src/lib/supabase/server.ts` ruft `createServerClient(...)` und gibt **bei jedem Aufruf eine neue Instanz** zurueck (kein Caching am createClient-Layer). React's `cache()` vergleicht Args per `Object.is` (===). Da Layout und Page je eine eigene `createClient()`-Instanz halten, sind die Args-Tupel `[supabase#1, tenantId]` und `[supabase#2, tenantId]` ungleich → Cache-MISS → die RPC `rpc_get_branding_for_tenant` wird weiterhin 2x pro Page-Load aufgerufen.

**Vitest Case 8 ist nicht repraesentativ** — der Test passiert nur die gleiche Mock-Supabase-Instanz an beide Calls und verifiziert damit nur dass cache() ueberhaupt funktioniert, nicht dass es im Production-Pattern (Layout+Page mit eigenen createClient-Aufrufen) greift.

**Impact:**

- **Production-Verhalten unveraendert:** weiterhin 2 RPC-Calls + 2 `rpc_get_branding_for_tenant`-Auswertungen pro Mandanten-`/dashboard`-Request (~5-10ms unnoetiger Overhead — identisch zur Pre-SLC-110-Baseline).
- **Kein Funktionsbruch** — die UI rendert korrekt, das Branding ist konsistent. Reine Performance-Optimierung schlaegt fehl.
- **Documentary-Gap:** ISSUE-049 ist in KNOWN_ISSUES.md mit "Code-Side Fix gepushed ... cache()-Wrap aktiv, Dedupe-Vitest Case 8 PASS" markiert, was suggeriert dass die Production-Wirkung erreicht ist. Sie ist es nicht.

**Loesungs-Optionen:**

1. **Quick-Fix Option A (~2 LoC)**: `createClient()` in `src/lib/supabase/server.ts` mit React `cache()` wrappen. Damit teilen sich Layout + Page + alle anderen Server-Component-Caller in derselben Render-Phase eine Instanz. Risiko: niedrig — cache() ist render-context-bound (faellt ausserhalb auf passthrough zurueck), Cookie-Store ist per-Request via `cookies()` ohnehin geteilt, kein Auth-State-Issue. Nebenwirkung: minimal positive Performance fuer alle anderen Doppelt-Caller im Code.

2. **Quick-Fix Option B (~5 LoC)**: `dashboard/page.tsx` Z.71 ersetzen — statt `resolveBrandingForTenant(supabase, profile.tenant_id)` den Wrapper `resolveBrandingForCurrentRequest()` aufrufen, plus `cache()`-Wrap um den Wrapper. Layout und Page rufen dann denselben gewrappten Wrapper → Cache-Hit per `cache()` mit 0 Args. Nachteil: Page muss profile-Lookup duplizieren oder akzeptieren (Wrapper macht eigene Profile-Query — minimal Overhead).

3. **Bigger Refactor**: `resolveBrandingForTenant` Signatur aendern auf `(tenantId)` only, internen `createClient()`-Aufruf (gewrappt). Bricht 12 Vitest-Cases + resolve-server-Caller. Cleanest aber teuer.

4. **Defer**: ISSUE-049 als "partial fix" re-opened in V6.1+-Polish. Akzeptiert dass die Performance-Optimierung verzoegert ist (hat keinen User-sichtbaren Effekt).

**Empfehlung:** Option A — minimaler Eingriff, korrekte Semantik, ~2 LoC + ein Vitest-Smoke. Kann direkt im naechsten /backend-Slot oder als kleiner Hotfix-Commit auf SLC-110 angehaengt werden.
