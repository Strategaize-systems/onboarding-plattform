## Problems found

1. **SMTP-ENV-Drift zwischen Slice-Spec und Codebase.** Slice-Spec sagt `SUPABASE_SMTP_HOST/PORT/USER/PASS`, der bestehende Code (`src/lib/email.ts`, `.env.deploy.example`) nutzt aber `SMTP_HOST/PORT/USER/PASS/FROM`. **Entscheidung:** existing Pattern uebernommen. `sendReminder` nutzt `SMTP_*`-ENVs konsistent mit `src/lib/email.ts`. Q-V4.2-I damit beantwortet: nodemailer direkt, da `@supabase/supabase-js` keine Custom-Send-Mail-API hat (nur Auth-Templates fuer Invite/Recovery).
2. **`captureUrl` Default ist `/dashboard`, nicht `/capture/start`.** Slice-Spec nennt als Beispiel `https://onboarding.../capture/start`, im Code zeigt der Reminder-Link auf `/dashboard`. Grund: `/capture/start` ist V3-Pfad fuer GF-Blueprint, employees sehen den employee-Workflow auf `/employee` aber Self-Service-Cockpit liegt auf `/dashboard`. Final-URL kann in /qa noch geaendert werden — kein Code-Issue, nur Routing-Klarstellung.
3. **`@/lib/logger` eager Supabase-Init crasht lokalen Build ohne ENVs.** Andere Routen mit Logger-Import (handbook/download, transcribe, etc.) haben das Issue bereits, aber unbemerkt weil sie pro-Build mit echten ENVs auf Coolify gebaut werden. Lokaler Build erfordert seitdem `SUPABASE_URL=stub`-Stub. **Empfehlung:** BL-XXX V4.3 — Lazy-Init-Pattern in `logger.ts` (getter statt eager const). Fuer SLC-048 in-scope: Tests mocken Logger.
4. **Test-Suite-Drift:** `npx vitest run` zeigt 147 Fail in 14 Files, aber alle wegen `TEST_DATABASE_URL is not set` (DB-Tests gegen Coolify-DB-Pattern, vorbestehend). Meine 19 neuen Tests laufen stand-alone gruen. Vor /qa SLC-048 Pflicht: `npm run test` auf Coolify-DB via node:20-Container ausfuehren (siehe Rule `coolify-test-setup.md`).
5. **PostgREST-Embed nicht ueber FK ohne tenant_admin-RLS:** Cron-Endpoint nutzt Service-Role explizit (RLS-Bypass), daher `tenants!inner(name)` auf employee_invitation funktioniert ohne RLS-Stolperer. Cross-Tenant-Lookup ist Pflicht-Funktion (Cron muss alle Tenants abdecken).

## Coolify-Cron-Setup-Anleitung (MT-7, Pflicht-Tabelle)

Pflicht-Voraussetzung VOR Cron-Anlage: ENV `CRON_SECRET` ist in Coolify gesetzt (32-byte hex), Container ist redeployed mit der neuen ENV.

| Feld         | Wert |
|--------------|------|
| **Name**     | `capture-reminders-daily` |
| **Command**  | `node -e "fetch('http://localhost:3000/api/cron/capture-reminders', { method: 'POST', headers: { 'x-cron-secret': process.env.CRON_SECRET } }).then(r => r.json()).then(console.log).catch(console.error)"` |
| **Frequency**| `0 9 * * *` (taeglich 09:00 UTC; Europe/Berlin entspricht 10:00 oder 11:00 je nach DST) |
| **Container**| `app` |

**Hinweise:**
- Container ist IMMER `app` (Coolify-Konvention fuer den Next.js-Container).
- Command nutzt `node -e fetch(...)` und NICHT `curl` — `curl` ist im Next.js-Image nicht installiert (memory `feedback_coolify_cron_node`).
- `localhost:3000` funktioniert ausschliesslich im Container.
- `CRON_SECRET` wird ueber `process.env` gelesen — der Wert lebt in Coolify-ENV, nicht im Command.
- Anlage **erst NACH erfolgreichem Live-SMTP-Test** mit Test-Mitarbeiter (MT-5). Sonst loesen die ersten Cron-Runs Mails aus, deren Spam-Reputation noch unverifiziert ist.
- Frequency `0 9 * * *` = 09:00 UTC = 10:00 Berlin (Winter) / 11:00 Berlin (Sommer). Wenn 09:00 lokal Berlin gewuenscht: `0 7 * * *` (Sommer) oder `0 8 * * *` (Winter) — Coolify hat keine Timezone-Option, Pattern lebt mit DST-Drift.
