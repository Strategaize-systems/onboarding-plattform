**Determinismus-Check** (zusaetzlich zur Spec-Risk-Auflistung):
- Cycle-Konsolidierung via `lex-min` ist deterministisch und unabhaengig vom Start-Knoten: bei Cycle {A, B} returnt der Walk von A wie auch der Walk von B beide "A". `for (const id of visited)` iteriert ueber JS-Set in Insertion-Order, aber das `< smallest`-Compare wirft Insertion-Order aus dem Output raus.
- Output-Sortierung `first_date asc + Tie-Break per root_message_id.localeCompare` ist deterministisch. NULL-Dates landen ans Ende. Identische first_date konvergieren auf lex-min root_message_id.
- Stable: zwei Walker mit demselben Input liefern identischen Output (Tests `5 Emails, lineare Kette`, `Star-Pattern`, etc. wuerden bei Non-Determinismus flaky werden — Vitest sieht keine Flakes).

**Edge-Case-Verifikation Forward-Parent-IS-Root (kritischer Bug-Fix aus RPT-394):**

- Das verifiziert, dass der Fix korrekt ist: parent IS Sub-Thread-Root (nicht current).
