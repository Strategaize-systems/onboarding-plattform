## Risk-Audit R2 (V5-Pipeline-Email-Spezialitaet)

Spec L230 R2: "V5-PII-Pipeline ist auf Walkthrough-Transkripte zugeschnitten — Email-spezifische Patterns (Signaturen, Email-Adressen-Headers) muessen via Adapter-Wrapper vor V5-Call rein. MT-5 ist Hauptrisiko: wenn Pre-Processing Patterns nicht ausreichen, leaken Klarnamen nach Sonnet (SLC-167). /qa Pflicht-Pattern-Scan auf redacted_body in MT-7."

| Sub-Risk | Code-Side-Mitigation | Status |
|---|---|---|
| Email-Signaturen werden nicht gestripped → Signatur-Namen leak | `stripSignature` mit 7 Trigger-Regexes (RFC-3676 + DE/EN-Gruesse) + 3-Folge-Zeilen-Drop. 7 Vitest-Cases GREEN | ✓ deterministische Pre-Processing |
| Email-Headers (From/To/Cc) werden mit-redacted statt durch Pseudonyme ersetzt | `buildEmailRedactUserMessage` rendert Headers als `From: P1` etc. — Adressen werden VOR LLM-Call ersetzt | ✓ deterministische Pre-Processing |
| Email-Adressen im Body-Text werden mit-redacted | `replaceParticipantsInBody` sortiert nach Laenge desc + case-insensitive Replace | ✓ deterministische Pre-Processing |
| **Klarnamen im Body-Text (z.B. "Max Mustermann") leaken** | Pre-Processing baut nur Email→Pseudonym-Map. Display-Names werden vom V5-LLM via Regel 9 im Email-Hint gefangen. **Reliability haengt von LLM-Output ab** | ⚠ Pre-Processing-Gap, Spec-akzeptiert + MT-7-Recall-Validation Pflicht |

**Detail zur Reliability-Gap (RPT-396 P-4 fortgesetzt):**

Pre-Processing ersetzt `alice@example.com` → `P1`. Wenn Body sagt `"Alice Schmidt"`, dann muss der LLM:
- Wissen, dass `alice@example.com` zu `P1` gehoert (System-Prompt-Liste vermittelt das)
- Vermuten, dass `Alice Schmidt` zu `alice@example.com` gehoert (Email-Prefix-Match)
- `Alice Schmidt` → `P1` ersetzen (Regel 9)

Reliability-Bereiche:
- **Hoch**: Email-Prefix = Vorname (`alice@x.com` ↔ `Alice ...`)
- **Mittel**: Nicknames, Initialen, Mid-Email-Prefixes
- **Niedrig**: Nur-Nachname-Referenzen (`Schmidt` allein in Body)

V9.0 akzeptiert das per Spec R2 mit Pflicht-Pattern-Scan in MT-7. SLC-167-Curation-UI gibt GF zusaetzlich eine manuelle Kontrolle.

**Ergebnis:** R2 Code-Side GREEN. Pre-Processing macht das deterministisch Mögliche; Klarnamen-Reliability ist Spec-akzeptierte LLM-Verantwortung mit MT-7-Defense.
